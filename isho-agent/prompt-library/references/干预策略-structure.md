# 干预策略

> 用途：指导 sub-agent 为每个用户构建和维护个性化的干预策略。
> 维护方：产品团队
> 参考：`睡眠第一性原理.md` 中的三件事（什么时候睡 / 睡眠质量 / 白天补充）和行为改变原理

---

## 一、策略是什么

一份好的干预策略 = 这个用户的精力全景分析 + 个性化的干预路线图 + 状态预案。

干预策略由 6 个 section 组成，分为三层：

```
原理层    [principles]        — 精力全景分析 + 策略路线图 + 状态预案
认知+行动层 [cognition] [action] — 当前在推什么、用户理解到了什么程度
记录层    [redlines] [history] [trends] — 边界、历史、数据
```

原理层驱动认知层和行动层：认知引导帮用户理解"为什么要这样做"，行动干预帮用户实际去做。两者互相强化——理解原因的用户执行力更强，执行后看到数据变化的用户理解更深。

---

## 二、如何构建个性化策略

### 2.1 精力全景分析

对每个用户，必须基于第一性原理的三件事，**逐一**分析这个用户的状态：

- **第一件事（什么时候睡）**：这个用户的作息规律性如何？有无错位？错位的原因是什么？
- **第二件事（睡眠质量）**：深睡够不够？三个条件（入睡时间/入睡前状态/环境）各是什么状态？
- **第三件事（白天补充）**：白天精力靠什么撑？有无反噬当晚睡眠的行为？

三件事不是只看"出问题的"——即使某个维度当前没问题，也要记录现状，因为用户状态会变化（生病、换工作、季节变化），需要知道哪些维度本来是好的。

### 2.2 策略路线图

基于全景分析，为这个用户规划干预路线：

- **当前焦点**：正在解决什么，为什么先解决这个
- **下一步**：当前焦点解决后，下一个要推的方向
- **远期方向**：更长远的改善空间（可能暂时不动但要记住）
- 路线图中每一步都要说清楚对应三件事中的哪一件

### 2.3 状态预案

为这个用户的可预见状态变化写好预案：

- **压力期预案**：根据这个用户的压力模式（如赶项目、周日焦虑），策略怎么调
- **生病/身体不适预案**：哪些干预暂停、哪些保留、恢复后怎么回来
- **这个用户特有的波动场景**：如加班日、聚餐日、周末——各自怎么处理

预案必须是个性化的（基于这个用户的画像），不是通用建议。

## 三、Section 数据结构定义

### [principles] — 精力全景 + 策略路线图 + 状态预案

**职责**：基于第一性原理三件事，对这个用户做全景分析，输出路线图和状态预案。这是整个策略的骨架——认知层和行动层都从这里出发。

**格式**：

```yaml
# [principles]
精力需求: {这个用户什么时候需要什么精力}

全景分析:
  什么时候睡:
    现状: {这个用户的作息规律性、错位情况}
    问题: {具体问题，如有} — {原因}
    可干预空间: {能动什么、不能动什么（红线约束）}
  睡眠质量:
    现状: {深睡水平、三个条件各自的状态}
    问题: {具体问题，如有} — {原因}
    可干预空间: {同上}
  白天补充:
    现状: {咖啡因/午睡/运动/光照的使用情况}
    问题: {具体问题，如有} — {原因}
    可干预空间: {同上}

策略路线图:
  当前焦点: {在解决什么} [关联:{三件事中的哪一件}]
  下一步: {当前焦点完成后的方向} [关联:{哪一件}]
  远期: {更长远的改善空间} [关联:{哪一件}]

状态预案:
  {这个用户特有的波动场景1}: {策略怎么调}
  {这个用户特有的波动场景2}: {策略怎么调}
  生病/身体不适: {暂停什么、保留什么、怎么恢复}
```

**填写规则**：
- 全景分析三件事都要写，即使某个维度当前没问题（写"现状正常"并说明为什么）
- "可干预空间"要考虑红线约束（如咖啡是红线则标注"不可从此切入"）
- 路线图每一步锚定到三件事中的具体维度
- 状态预案从画像的 `[routines]` `[psychology]` `[lifestyle]` 提炼，不写通用建议
- 当用户状态发生重大变化时（换工作、搬家、生病）整体更新

---

### [cognition] — 认知维度

**职责**：跟踪用户对第一性原理的认知状态，规划学习路径，定义引导触发器。

**格式**：

```yaml
# [cognition]
已建立:
  - {用户已理解的认知点} [关联:{这个认知对应三件事中的哪一件}] {可选：行为层面的状态}
待建立:
  - {需要引导的认知点} [关联:{对应哪一件}] ← 优先级:{高/中/低}, {为什么重要}
引导策略:
  - {整体引导方式偏好}
  - 触发器: {当X场景出现时} → {用Y方式引导Z认知}
  - 触发器: {当X场景出现时} → {用Y方式引导Z认知}
```

**填写规则**：
- `已建立` 条目必须标注关联到三件事中的哪一件（如 `[关联:入睡前状态]`、`[关联:作息时间匹配]`）
- 可附加行为状态："但做不到停"说明认知已有但行动未跟上，此时需要行动层而非认知层解决
- `待建立` 按优先级排序，优先级依据：直接影响当前干预方向 > 影响未来干预方向 > 一般性知识
- 引导触发器格式：`触发器: {场景条件} → {引导方式和目标认知}`
- 场景条件应具体可判断（"周末补觉数据出现时"而非"合适的时候"）
- 认知从"待建立"移到"已建立"的条件：用户在对话中主动使用该概念，或用户的行为决策体现了该认知
- 用户多次对某个认知引导无反应时，降低优先级，不反复推

---

### [action] — 行动维度

**职责**：管理当前干预方案和用户的行动偏好。合并原 `[active]` 和 `[preferences]`。

**格式**：

```yaml
# [action]
当前干预:
  名称: {干预名称}
  在解决什么: {这个行动在解决三件事中的哪一件的什么问题}
  措施: {具体做什么、什么时间、怎么做}
  状态: {试跑中/执行中/固化中}({起始日期}), {效果概述}
  阻力: {执行中的困难}
偏好:
  接受: {用户接受的干预类型}
  不接受: {用户不接受的干预类型}
  原则: {用户偏好的推进方式}
下一步:
  有效路径: {如果当前干预有效，下一步做什么}
  无效路径: {如果当前干预无效，换什么方向}
```

**填写规则**：
- `在解决什么` 必须说清楚这个行动在解决 `[principles]` 中的哪个问题，让每个行动都有原理支撑
- `状态` 三阶段：试跑中（刚开始，<7天）→ 执行中（有初步数据）→ 固化中（稳定有效，正在养成习惯）
- `阻力` 记录具体的执行障碍，为调整方案提供依据
- `偏好` 从干预历史和用户反馈中提炼，不从数据推测
- `下一步` 必须有两条路径（有效/无效），避免走到死胡同
- 当前无活跃干预时，`当前干预` 标注"无"，保留偏好和下一步方向

---

### [redlines] — 边界

**职责**：记录用户明确拒绝的方向和需要谨慎对待的话题。

**格式**：

```yaml
# [redlines]
硬红线(用户明确拒绝):
  - {方向}: "{用户原话}" ({日期})
软约束(建议谨慎):
  - {话题}: {为什么需要谨慎}
```

**填写规则**：
- 硬红线必须保留用户原话和日期
- 硬红线永不删除
- 软约束可根据用户态度变化调整
- 红线与策略方向冲突时（如限制咖啡是红线但白天精力补充需要调整咖啡用量），在 `[principles]` 的可干预空间中标注"不可从此切入"

---

### [history] — 干预历史

**职责**：记录过去的干预尝试和原理级别的学习。

**格式**：

```yaml
# [history]
- {干预名称}({日期范围}): {结果概述}
  学习: {这次尝试对策略的启示}[{在解决什么问题}]
```

**填写规则**：
- 每条包含结果 + 学习两部分
- `学习` 标注在解决三件事中的什么问题，让历史教训与原理体系连接
- 学习内容应对未来策略有指导意义（"环境设计有效"而非仅"做到了"）
- 最多保留 10 条，超出删除最旧的
- 从 `[action]` 移出的干预自动进入此 section

---

### [trends] — 数据趋势

**职责**：近期关键指标的变化方向，供主 agent 用数据说话。

**格式**：

```yaml
# [trends]
{时间范围}:
  {指标}: {数值变化} ({变化方向/幅度})
干预日 vs 非干预日:
  {指标}: {对比数值} (差{数值})
```

**填写规则**：
- 使用绝对日期，禁止相对时间词
- 只记录有意义的变化（>5% 或有趋势方向），不逐日列举
- 干预日 vs 非干预日的对比是评估当前干预效果的核心数据

---

## 四、完整示例

```yaml
# [principles]
精力需求: 工作日朝十晚七, 需要稳定的白天精力

全景分析:
  什么时候睡:
    现状: 工作日01:30入睡/08:45起床(~7h窗口,实睡6.5h); 周末02:30入睡/10:30起床, 社交时差~2h
    问题: 工作日周末作息差太大 — 每周一相当于倒一次时差, 周一精力评分低15%
    可干预空间: 起床端可控(工作日已稳定); 周末起床时间可尝试收窄但用户有顾虑(软约束)
  睡眠质量:
    现状: 入睡潜伏期正常(~15min), 夜醒极少; 深睡偏低(18%, 正常20-25%)
    问题: 入睡前没有过渡 — 刷手机1.5-2h直接入睡, 深睡与屏幕时间相关(r=0.72)
    可干预空间: 环境设计对此用户有效(手机放客厅已验证); 意志力型方案不接受
  白天补充:
    现状: 下午一杯咖啡(刚需/红线); 几乎不运动; 无午睡习惯; 遮光窗帘已装但无晨光暴露
    问题: 无显著问题, 但缺乏运动和光照是远期改善空间
    可干预空间: 咖啡不可动(红线); 早起运动不可动(红线); 晨光暴露可探索(非早起, 是拉窗帘)

策略路线图:
  当前焦点: 建立睡前过渡(手机放客厅→wind-down routine) [关联:睡眠质量]
  下一步: 收窄周末作息差(控制周末起床时间) [关联:什么时候睡]
  远期: 引入晨光暴露、轻度运动 [关联:白天补充]

状态预案:
  加班日(周三固定+偶发): 放宽执行要求, 不追责; 补偿性娱乐需求强, 不在当晚施压
  周五聚餐日: 不干预当晚作息; 聚餐后不建议立即执行手机放客厅, 允许自然入睡
  赶项目期(如4月中deadline): 暂停推新干预, 仅维持已固化习惯; 期间数据波动不做评判
  周日焦虑: 不在周日晚给新建议; 已固化的习惯照做, 焦虑本身不是干预目标
  生病/身体不适: 暂停所有干预, 允许多睡; 恢复后从手机放客厅开始恢复, 不加新的

# [cognition]
已建立:
  - 睡前手机影响入睡 [关联:入睡前状态] (知道但做不到停→行动层解决)
  - 深睡对精力的重要性 [关联:睡眠质量]
待建立:
  - 工作日周末作息差大相当于每周倒时差 [关联:作息时间匹配] ← 优先级:高, 直接影响周末行为决策
  - 酒精会减少深睡 [关联:睡眠质量] ← 优先级:中, 周五聚餐场景相关
引导策略:
  - 用他自己的数据做前后对比，不讲大道理
  - 触发器: 周末补觉数据出现时 → 用工作日vs周末深睡对比，让用户看到作息差带来的影响
  - 触发器: 周五聚餐后深睡数据明显差时 → 用当晚vs前一晚数据，让用户看到酒精对深睡的影响

# [action]
当前干预:
  名称: 手机放客厅
  在解决什么: 入睡前没有过渡时间(睡眠质量→入睡前状态)
  措施: 每晚23:00闹钟→手机放客厅充电
  状态: 试跑中(2026-03-23开始), 执行日入睡提前45min/深睡+3%
  阻力: 加班日(周三+偶发)"补偿性娱乐"需求强
偏好:
  接受: 环境改变、定时提醒、渐进式目标
  不接受: 需意志力的("忍住不刷手机")、大幅改作息的
  原则: 先小范围试跑, 数据证明有效再固化
下一步:
  有效路径: 固化→提前到22:30→引入轻度过渡活动(拉伸/阅读)
  无效路径: 尝试手机定时锁屏工具辅助

# [redlines]
硬红线(用户明确拒绝):
  - 限制咖啡: "下午必须靠咖啡撑着" (2026-03-18)
  - 早起运动: "早上根本起不来" (2026-03-15)
软约束(建议谨慎):
  - 周五社交: 用户重视社交, 不要频繁建议减少
  - 饮酒话题: 用户表示不想讨论

# [history]
- 手机放客厅(03-20~03-22): 部分有效, 执行日入睡提前45min/深睡+3%, 加班日做不到
  学习: 环境设计对此用户有效[入睡前状态], 但加班日的补偿心理是独立阻力
- 限制咖啡(03-18): 拒绝→红线
  学习: 咖啡是用户刚需, 白天精力补充不能从限制咖啡切入
- 早起运动(03-15): 拒绝→红线
  学习: 用户是晚型人, 早起逆身体节奏, 不应从起床端施加压力

# [trends]
本周(W13) vs 上周(W12):
  平均入睡: 01:15 → 01:05 (提前10min)
  平均时长: 6.5h → 6.8h (+18min)
  深睡占比: 16% → 18% (+2%)
  HRV均值: 42ms → 45ms (+3ms)
干预日 vs 非干预日(本周):
  入睡: 00:45 vs 01:25 (差40min)
  深睡: 21% vs 16% (差5%)
```
