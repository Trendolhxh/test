# 睡眠画像初始化

## Context

需要基于用户的健康数据生成或更新用户画像。
这是画像生命周期的起点——只做数据能直接得出的事，不做任何提问和推测。
所有需要用户确认的字段统一标注 [待确认]，由下游的 sleep-profile-complete skill 负责补充。

## Prerequisites

工作目录下有 references/用户画像-structure.md 作为画像结构模板。
如果是更新场景，用户画像.md 已存在。

## Instructions

### 1. 拉取健康数据

用 show_status 告知用户正在分析数据。

通过 get_health_data 拉取近 3-6 个月的多维度数据，尽量在一次调用中覆盖所有相关指标：

**睡眠**：睡眠分期（深睡/REM/浅睡/清醒）、睡眠负债、睡眠一致性。重点关注就寝和起床时间的均值与标准差、工作日 vs 周末差异。

**心血管**：静息心率、HRV（SDNN）。关注趋势方向——是在改善还是恶化。

**血氧**：SpO2 分布，低于 95% 的频率。

**活动**：步数、运动记录、日照时间。

**体征**：呼吸频率等可用指标。

同时通过 get_user_profile 获取已有的用户基础信息（如果有的话），避免覆盖已知信息。

### 2. 分析数据，提炼画像要素

不是把原始数据搬进画像，而是从数据中提炼出有画像价值的信息：

- **核心指标的均值、分布和波动幅度**——均值只是起点，标准差和极值更能刻画用户特征
- **结构性模式**——工作日/周末差异、补觉模式、就寝时间规律性
- **月度趋势**——哪些指标在改善、哪些在恶化、哪些平稳
- **异常信号**——值得关注但不下诊断的数据点（如反复出现的血氧偏低）
- **积极信号**——用户的优势和能力（如"给够时间时深睡表现正常"）

### 3. 生成或更新用户画像.md

读取 references/用户画像-structure.md 了解完整结构，按板块填写。

**能从数据填写的板块**：基本信息（有数据的部分）、睡眠数据（核心指标表+趋势表，这是最丰富的板块）、身体状况（仅记录异常信号，不推断疾病）、运动数据、典型作息（从就寝起床时间推断框架，标注"基于数据推断，待用户确认"）。

**整体标注[待确认]的板块**：核心诉求、饮食、睡眠环境、已有认知与实践、干预偏好与约束、干预历史。

每条信息的格式：`{内容} [数据]` 或 `[待确认]`。不允许出现没有来源标注的信息，不允许出现 [自述] 标注。

如果是更新已有画像，只更新数据相关板块，不覆盖已有的 [自述] 内容。

### 4. 向用户展示数据摘要

展示关键发现——基本信息、睡眠核心数据、值得关注的异常、积极的趋势。
明确告诉用户哪些板块还是空的，需要通过后续对话补充。

## Core Principles

**信息必须有出处**：每条写入画像的信息标注 [数据]。本 skill 没有 [自述] 来源——所有非数据信息都写 [待确认]。

**认知不能猜**：数据只能告诉你身体指标和行为模式，不能告诉你用户知道什么、关心什么。认知和诉求板块整体标注 [待确认]。

**不生成建议**：本 skill 只记录数据事实，不给任何建议。如实记录异常，为下游生成策略提供依据。

## Output Format

输出文件：`用户画像.md`，保存到用户的工作目录。

展示给用户的摘要：
```
你的健康数据已分析完成，画像[初版已生成 / 已更新]。几个关键数据：

- [基本信息摘要]
- [睡眠核心数据——均值+波动+结构性模式]
- [值得关注的异常信号]
- [积极的趋势]

目前还有 [N] 个板块需要通过对话补充（[具体板块名]）。
准备好的话我们可以开始聊。
```

## Guardrails

- 不从睡眠数据推断用户的睡眠困扰——只写数据事实"深睡占比 X%"，困扰留给用户自述
- 不从行为数据推断偏好——只写数据"周末起床均值 X:XX"，偏好留待确认
- 不把"没有数据"当作"没有"——某项数据缺失不等于用户没有该情况，写 [待确认]
- 不在画像中夹带建议——发现血氧偏低只记录数据，不写"建议排查呼吸问题"
- 更新已有画像时不覆盖 [自述] 内容——数据刷新数据板块，用户自述的信息保持不变
