# 更新用户画像

## Context

需要基于用户的健康数据生成或更新用户画像。
这是画像生命周期的数据层——只做数据能直接得出的事，不做任何提问和推测。
所有需要用户确认的字段统一标注 [待确认]，由下游的 sleep-profile-complete skill 负责补充。



## Instructions

### 1. 先读画像，再决定拉多少数据
通过 get_user_profile 获取已有的用户基础信息，判断数据板块的状态，避免覆盖已知内容：

- **画像为空或首次生成**：需要拉取长周期数据（最长 3 个月）来建立基线
- **画像已有基线，需要更新近期变化**：只需要拉取自上次更新以来的数据
- **画像基本完整，只是缺少最新一晚的数据**：可以只拉最短时间窗口（短至 30 分钟级别的今日数据）

数据拉取的量由画像当前缺什么决定，不是每次都全量刷新。

### 2. 拉取健康数据

通过 get_health_data 拉取所需时间窗口的数据。拉取原则：

- **覆盖画像结构中所有数据相关板块**——不要只拉睡眠，心血管、活动、体征等凡是工具支持的指标都应该覆盖
- **一次调用尽量多带指标**——减少调用次数，相关指标合并请求
- **时间窗口匹配需求**——首次建基线用长周期，日常更新用短周期，不多拉不少拉
- **数据项可能随时间增加**——按画像结构模板中定义的板块来决定该拉什么，而不是依赖一个固定的指标列表

同时通过 get_user_profile 获取已有的用户基础信息，避免覆盖已知内容。

### 3. 分析数据，提炼画像要素

不是把原始数据搬进画像，而是从数据中提炼出有画像价值的信息：

- **核心指标的均值、分布和波动幅度**——均值只是起点，标准差和极值更能刻画用户特征
- **结构性模式**——数据中跟生活节奏绑定的规律
- **趋势方向**——哪些指标在改善、哪些在恶化、哪些平稳
- **异常信号**——值得关注但不下诊断的数据点
- **积极信号**——用户的优势和能力

### 4. 生成或更新用户画像.md

读取 references/用户画像-structure.md 了解完整结构，按板块填写。

能从数据填写的板块标注 `[数据]`。不能从数据得出的板块标注 `[待确认]`。
不允许出现没有来源标注的信息，不允许出现 [自述] 标注。

更新已有画像时，只刷新数据相关板块，不覆盖已有的 [自述] 内容。

静默完成，不向用户展示结果。画像更新是后台操作，下游 skill 或对话流程会按需读取画像内容。

## Core Principles

**信息必须有出处**：写入画像的每条信息标注 [数据]。本 skill 没有 [自述] 来源。

**认知不能猜**：数据只能告诉你身体指标和行为模式，不能告诉你用户知道什么、关心什么。

**不生成建议**：只记录数据事实，不给建议。如实记录异常，为下游提供依据。

## Guardrails

- 不从数据推断用户的主观感受或偏好——数据事实和用户认知是两回事
- 不把"没有数据"当作"没有"——数据缺失写 [待确认]
- 不在画像中夹带建议
- 更新时不覆盖 [自述] 内容

## Examples

<example>
场景：新用户，画像不存在

→ 画像为空，需要建基线 → 决定拉取 3 个月数据
→ 按画像结构模板覆盖所有数据相关指标
→ 拉取已有用户基础信息（无）
→ 分析：提炼均值/波动/结构性模式/趋势/异常/积极信号
→ 生成 用户画像.md，数据板块填入 [数据]，其余标注 [待确认]
→ 静默完成
</example>

<example>
场景：画像已存在，上次更新 2 小时前，新的一晚睡眠数据已到

→ 读取画像：基线已有，数据板块上次更新 2 小时前
→ 只需要最新一晚的数据 → 拉取 today 范围
→ 对比新数据与画像中的基线，更新有变化的指标
→ 发现深睡占比连续 3 天下降 → 更新趋势方向
→ 保留所有 [自述] 内容不动，只刷新数据板块
→ 静默完成
</example>
