# 给出行为干预建议

## Context

对话中已经有了分析结论或用户明确的求助，需要把洞察转化为一个可执行的行动。
本 skill 不从零开始分析——它依赖上文已有的信息来决定建议什么方向、什么力度。

## Instructions

### 1. 明确上文给出的建议场景

先梳理上文已有的信息，搞清楚两个问题：

- **在什么情况下给建议**：是睡眠分析发现了节律漂移？是用户说"最近总是睡不着"？是反馈卡显示方案需要调整？不同的触发场景决定建议的方向和力度。
- **应该给什么方向的建议**：上文的分析结论指向什么？用户的诉求是什么？这两者是否一致？

把这个场景判断带入下一步，而不是脱离上文凭空给建议。

### 2. 拉取干预策略

通过 get_strategy 加载用户的红线、当前干预方案和干预历史。

建议必须有延续性：
- 有活跃方案 → 建议是对现有方案的微调、强化或推进下一步，而不是另起炉灶
- 有干预历史 → 参考过去哪些方向有效、哪些失败过，不重复走老路
- 没有历史 → 从上文分析中最明确的发现出发，提出第一步

红线不可触碰——用户明确拒绝过的方向不建议。

### 3. 给一个建议

只给一个。具体到可以执行——有明确的行动、时间点和做法。
建议必须锚定在上文的分析发现或用户诉求上，让用户看到"为什么是这个建议"。

如果不确定该推荐什么方向，坦诚说不确定，给 2-3 个选项让用户自己选。

### 4. 告知 check 时间，确认用户接受

告诉用户你会在什么时候来跟进（通常是第二天早上），然后用 send_feedback_card 发一张确认卡片，问用户是否接受这个建议。

卡片选项应该包含：接受、换一个建议。
用户选择换建议时，回到 Step 1 重新评估方向。

## Output Format

```
[1-2 句话点明建议的依据——关联上文分析或用户诉求]
[一个具体建议：行动 + 时间点 + 做法]
[告知 check 时间]
[确认卡片：接受 / 换一个]
```

## Guardrails

- 上文没有分析依据时不给建议——先做分析
- 不一次给多条建议
- 不触碰红线

## Examples

<example>
场景：上文睡眠分析发现工作日入睡偏晚（均值 00:54），周六给够时间深睡正常，
用户当前方案是"手机放客厅"已执行一周

→ 上文结论：问题在工作日时间窗口不够，不是睡眠能力问题
→ 拉取策略：手机放客厅已稳定，红线里没有涉及就寝时间调整

output:

你手机放客厅已经稳了一周，数据也有在响应。现在瓶颈在工作日上床太晚——
平均 00:54，深睡时间被压缩了。

建议在手机放客厅的基础上加一步：工作日 00:15 前上床。
具体做法是把睡前拉伸提前到 23:50 开始，00:15 躺下。

明天早上我来问你执行情况。

→ send_feedback_card: "今晚试试 00:15 前上床？"
  选项：接受 / 换一个建议
</example>

<example>
场景：用户主动问"有什么办法能睡好一点"，没有前置分析

→ 上文没有分析结论，只有用户诉求
→ 拉取策略：当前无活跃方案，历史里试过褪黑素（无效）、睡前冥想（拒绝）

output:

你之前试过褪黑素没什么效果，冥想也不想做——了解。
现在还不太确定哪个方向最适合你，先看一下最近的数据再给建议。

→ 转入 sleep-data-review skill 做分析，分析完后再回到本 skill
</example>
