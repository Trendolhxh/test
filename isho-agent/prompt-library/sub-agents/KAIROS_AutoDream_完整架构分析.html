1. KAIROS 模块全景概览

KAIROS 这个名字来自古希腊语 καιρός，意为"恰当的时机"。在泄露的 Claude Code 源码中，KAIROS 是一个被提及超过 150 次的功能标志，代表着从"被动响应式 AI 工具"到"始终在线的自主守护进程"的范式转变。

KAIROS 的核心定位

KAIROS 不是一个简单的后台任务。它是一个持续运行的自主 Agent，能够：

  • 在用户不操作时持续观察工作环境
  • 维护每日追加写入的观察日志
  • 通过 <tick> 心跳机制定期唤醒并决定是否主动行动
  • 在用户空闲时执行 AutoDream 记忆整合
  • 受 15 秒阻塞预算约束，绝不打断用户工作流

2. KAIROS 系统架构与核心组件

2.1 Tick 心跳机制

KAIROS 的核心驱动力是 <tick> 心跳提示。系统定期向 Agent 发送包含用户当前本地时间的 tick 消息，Agent 收到后决定是否需要采取行动。

Tick 消息的设计要点
  • 含义：每个 tick 等于 "你醒着呢，现在做什么？"
  • 时间信息：tick 中的时间是用户本地时间，用于判断时段（工作时间 vs 深夜）
  • 时区注意：外部工具（Slack、GitHub 等）的时间戳可能在不同时区
  • 行为选择：Agent 可以选择主动行动或保持安静
  • 成本意识：每次唤醒都消耗一次 API 调用

2.2 Sleep 工具与节奏控制

KAIROS 通过 Sleep 工具控制两次行动之间的等待时间：等待慢进程时睡久一点，主动迭代时睡短一点。关键约束是 prompt cache 在 5 分钟无活动后过期——需要在 API 调用成本和缓存命中率之间找平衡。

2.3 15 秒阻塞预算

阻塞预算：不打断用户的硬约束

任何会阻塞用户工作流超过 15 秒的主动行为都会被延迟执行。这是"有用但不烦人"的设计哲学的工程化实现。相当于一个安静的同事在背后默默工作，只在合适时机轻声提醒。

2.4 Brief 输出模式

当 KAIROS 活跃时，会启用 Brief 输出模式——极度简洁的响应，专为持续运行的助手设计，避免在终端中产生刷屏式输出。

2.5 Daily Log 日志系统

KAIROS 维护追加写入（append-only）的每日日志文件，全天记录观察、决策和行动。这些日志文件随后成为 AutoDream 记忆整合的原始素材。

4. AutoDream 记忆整合引擎

4.1 设计灵感：模拟人类 REM 睡眠

AutoDream 的命名是刻意的类比：就像人类大脑在 REM 睡眠期间将短期记忆整合为长期存储一样，AutoDream 在用户空闲时对 Agent 的记忆文件执行"反思性整理"。这不是简单的数据清理，而是一个 知识蒸馏 过程。

4.2 三重门控触发机制

三重门控：防止过度整合和不足整合
门控条件设计意图
时间门距上次 Dream ≥ 24 小时避免频繁整合浪费资源
会话门距上次 Dream ≥ 5 个会话确保有足够新信息值得整合
锁门获取整合锁 (防并发)防止多个 Dream 进程同时运行

三个条件必须全部满足。例如：一个持续两天的长会话不会触发（会话数不够）；两小时内跑了 10 个快会话也不会触发（时间不够）。

4.3 执行方式：Fork 子 Agent

AutoDream 作为一个 Fork 子 Agent 运行——创建父 Agent 上下文的完整副本后在独立进程中执行。这样做有两个关键好处：

  • 不污染主循环：记忆整合的"思维链"不会干扰主 Agent 的工作
  • 命中 Prompt Cache：复制的上下文可以利用 API 的 prompt cache，降低成本

4.4 四阶段整合流程

Phase 1 — Orient（定向扫描）
  • ls 记忆目录，了解当前有哪些文件
  • 读取 MEMORY.md（索引文件），理解当前索引结构
  • 浏览现有专题文件（topic files），避免后续创建重复内容
  • 如有近期日志文件，一并审阅
Phase 2 — Gather Signal（收集信号）
  • 搜索近期会话记录（JSONL 格式的 transcript 文件）
  • 使用 grep 窄搜索，而非完整读取——降低 token 消耗
  • 重点搜索的高价值模式：
    • 用户纠正（说明之前的记忆有误）
    • 明确的保存请求（"记住这个"）
    • 反复出现的主题和模式
    • 关键决策和偏好变更
Phase 3 — Consolidate（整合写入）
  • 将新发现合并到现有专题文件中，而非创建近似重复的新文件
  • 将相对日期转为绝对日期（"昨天" → "2026-03-30"），确保时间推移后仍可理解
  • 删除被推翻的事实——如果今天的调查推翻了旧记忆，在源头修正
  • 移除对不存在文件的引用
  • 合并重叠条目，消除逻辑矛盾
  • 将模糊见解转化为确定事实
Phase 4 — Prune & Index（修剪索引）
  • 重建 MEMORY.md 为精简索引，严格控制在 200 行以内
  • 每个索引条目一行，约 150 字符：标题 + 一句 hook
  • 绝不将记忆内容直接写入索引——索引只存指针
  • 移除指向过时记忆的指针
  • 将冗长条目降级到专题文件中
  • 添加指向新重要记忆的指针
  • 解决索引中的矛盾
性能参考：在一个观测案例中，AutoDream 在不到 9 分钟内整合了 913 个会话的记忆。

8. 产品设计启示

8.1 KAIROS 模式的核心设计智慧

① "有用但不烦人"的工程化

15 秒阻塞预算 + Brief 输出模式 + Sleep 工具节奏控制 = 一个不会打扰你但始终在工作的助手。这是对"主动 Agent"最大用户体验风险（打断工作流）的精确工程化解法。

② 心跳驱动 vs 事件驱动

KAIROS 选择了 tick 心跳而非事件驱动。这意味着 Agent 是"定期醒来检查"而非"等待触发"。好处是 Agent 能主动发现问题，坏处是有持续成本。通过 Sleep 工具动态调节心跳间隔来平衡。

③ 记忆整合 = 离线批处理

AutoDream 本质上是一个离线批处理 ETL 任务：Extract（从 JSONL 中 grep 信号）→ Transform（合并、去重、转日期、消矛盾）→ Load（写入专题文件 + 重建索引）。这个模式可以直接复用到任何需要长期记忆的 Agent 产品中。

④ 三重门控：防止资源浪费的精确控制

时间门 + 会话门 + 锁门的三重门控设计非常优雅：用最少的逻辑防止了过度整合（浪费 API）和不足整合（记忆退化）。任何需要后台定时任务的 Agent 都应该考虑类似的多条件门控。
