# 首页数据同步状态 (Homepage Data Sync Status) PRD

> 文档版本：v1.0 | 创建日期：2026-04-10
> 状态：**正式 PRD**
> 所属模块：首页 · 数据同步状态层

---

## 一、功能定义

### 1.1 核心定义

用户每次打开 App 时，系统自动从穿戴设备同步最新健康数据。首页顶部显示全局同步状态指示，所有数据卡片作为一个整体同步：

1. **同步中**：顶部显示同步状态指示，数据区域展示上次缓存值，用户可正常阅读旧数据
2. **同步成功**：顶部状态指示消失，数据平滑过渡到最新值（CountUp 动画），给用户明确的"已更新"感知
3. **同步失败/数据过期**：顶部显示原因提示条 + 重试按钮，数据区域展示上次缓存值

### 1.2 与 Smart Digest 的关系

| 维度 | Smart Digest | 数据同步状态 |
|:---|:---|:---|
| 定位 | AI 洞察层（数据解读） | 数据管道层（数据新鲜度） |
| 数据依赖 | 依赖同步完成后的健康数据 | 监控同步过程本身 |
| 触发时机 | 同步完成 + 冷却期 + 数据有变化 | 每次打开 App 即刻触发 |
| 失败表现 | 摘要为空，数据照常 | 数据区域显示状态指示 |

**关键衔接**：数据同步状态位于 Smart Digest 的**上游**。全局同步成功后才触发 Smart Digest 生成，确保首页数据和 AI 摘要基于同一套完整数据。同步失败或数据过期时，Smart Digest 不重新生成，展示上次缓存的摘要。

### 1.3 设计理念

| 原则 | 说明 |
|:---|:---|
| **透明不焦虑** | 告诉用户"在等什么"，而不是让页面空白令人困惑 |
| **归因到设备，不归因到 App** | 原因文案指向穿戴设备状态（未连接/未同步），不暗示 App 有 bug |
| **非阻塞** | 同步状态是信息层，不拦截用户浏览缓存数据或使用对话功能 |
| **自动消退** | 同步成功后状态指示器自动消失，无需用户操作 |
| **只说能确定的事** | 基于系统实际可检测的信息设计文案，不猜测无法确认的原因 |

## 二、同步状态

### 2.1 状态定义

| 状态 | 含义 |
|:---|:---|
| 同步中 | 正在从穿戴设备拉取最新健康数据 |
| 同步成功 | 数据拉取完成，有新数据更新到页面 |
| 同步成功（无变化） | 数据拉取完成，但与上次一致，无需更新 |
| 数据过期 | 同步失败、超时或设备数据不够新鲜 |

### 2.2 状态流转

- 用户打开 App → 进入"同步中"
- 同步完成且数据新鲜 → "同步成功"或"同步成功（无变化）"
- 同步超时 / 出错 / 数据超过 12 小时未更新 → "数据过期"
- 用户点击重试或下次打开 App → 重新进入"同步中"

### 2.3 全局同步

所有健康数据作为一个整体进行同步，页面维护一个全局同步状态（而非每张卡片独立状态）。任一数据查询失败或数据过期，全局状态即进入"数据过期"。

首页涉及的数据类型：

| 数据卡片 | 同步方式 |
|:---|:---|
| 睡眠评分 | 直接从穿戴设备同步 |
| 心率 | 直接从穿戴设备同步 |
| 消耗 | 直接从穿戴设备同步 |
| 高能时段 | 依赖睡眠 + 心率数据，由后端计算 |
| 入睡窗口 | 依赖睡眠历史数据，由后端计算 |
| 睡眠分析 | 直接从穿戴设备同步 |

> 所有设备数据并行查询，全部返回后再触发后端计算。设备查询 + 后端计算全部完成后，全局状态才从"同步中"转为"同步成功"。

## 三、失败原因识别与用户文案

### 3.1 多设备来源支持

用户的穿戴设备不一定是 Apple Watch，可能是小米手环、佳明手表、华为手环等第三方设备。App 需识别用户的主要数据来源设备，以便在同步失败时给出针对性的提示文案。

| 设备类型 | 特点 |
|:---|:---|
| Apple Watch | 系统可检测手表连接状态和蓝牙状态，能给出精确的失败原因 |
| 第三方设备 | 仅能检测数据是否过期，无法检测设备连接状态，只能引导用户打开伴侣 App |
| 无设备 | 无数据，引导用户连接穿戴设备 |

### 3.2 失败原因文案（按优先级排序）

当多个问题同时存在时，优先展示用户最可操作的一条。

#### 通用场景（所有设备适用）

| 优先级 | 场景 | 用户文案 | 用户操作 |
|:---:|:---|:---|:---|
| 1 | 用户未授权健康数据权限 | apple health 数据未授权 | 点击跳转系统设置 |
| 2 | 无数据且无设备（新用户） | apple health 数据未更新 | 新用户引导 |

#### Apple Watch 用户

| 优先级 | 场景 | 用户文案 | 用户操作 |
|:---:|:---|:---|:---|
| 3 | 未检测到已配对的 Apple Watch | 未检测到 apple watch | 引导配对 |
| 4 | iPhone 蓝牙已关闭 | 蓝牙已关闭 | 提示开蓝牙 |
| 5 | 手表已配对但无法连接 |  apple watch 连接失败 | 列出可能原因，不断言 |
| 5b | 手表已连接但无历史数据（首次配对） | apple watch 已连接 | 告知预期，减少焦虑 |

#### 第三方设备用户

| 优先级 | 场景 | 用户文案 | 用户操作 |
|:---:|:---|:---|:---|
| 3 | 数据超过 12 小时未更新 | apple health 数据未更新 | 引导打开设备 App |

#### 兜底

| 优先级 | 场景 | 用户文案 | 用户操作 |
|:---:|:---|:---|:---|
| 6 | 以上条件均不匹配（如超时、数据过期但设备状态正常等） | 数据同步中... | 自动重试 |

### 3.3 文案设计原则

| 原则 | 说明 |
|:---|:---|
| **格式** | `{现状描述}，{用户可做的事}` |
| **长度** | 不超过 25 字 |
| **禁止** | 不使用技术术语、不暗示 App 有 bug、不断言无法确认的原因 |
| **优先级** | 能跳设置 > 能开蓝牙 > 能检查设备 > 通用兜底 |
| **设备名称** | 第三方设备文案中使用实际的伴侣 App 名称（如"小米运动"） |

## 四、UI 设计

### 4.1 首页布局集成

同步状态以全局指示条的形式显示在页面顶部（问候语上方），不影响 Smart Digest 和数据卡片的布局：

```
┌─────────────────────────────────────────┐
│                                         │
│  ┌──────── 全局同步状态 ─────────┐       │  ← 同步中 / 数据过期时显示
│  │ 🔄 数据同步中...              │       │     同步成功后自动消失
│  └───────────────────────────────┘       │
│                                         │
│  早上好                                  │  ← 问候语（不受同步状态影响）
│  {digest or empty}                      │  ← Smart Digest（独立逻辑，不变）
│                                         │
│  ┌────┐  ┌────┐  ┌────┐                │
│  │睡眠│  │心率│  │消耗│                  │  ← 数据指标卡片
│  │ 92 │  │ 87 │  │672 │                  │
│  └────┘  └────┘  └────┘                │
│                                         │
│  [高能时段] [入睡窗口] [睡眠分析]        │  ← 功能卡片
│                                         │
│  [快捷提问 ①] [快捷提问 ②] [快捷提问 ③] │  ← Smart Digest 快捷提问（不变）
│  ┌─────────────────────────────────┐    │
│  │ ✨ 输入您的内容             ⬆   │    │  ← 输入框（不变）
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
```

### 4.2 各状态的 UI 表现

| 状态 | 顶部全局状态指示 | 数据卡片区域 |
|:---|:---|:---|
| 同步中 | 显示同步状态条（loading + "数据同步中..."） | 展示缓存值，用户可正常阅读 |
| 同步成功 | 状态条自动消失 | 数值动画更新至最新值（CountUp） |
| 同步成功（无变化） | 状态条自动消失 | 静态显示当前值，无动画 |
| 数据过期 | 显示原因 + 重试按钮 | 展示缓存值 + 灰色更新时间标注 |

> **有缓存时**：无论同步成功还是失败，首页始终有数据可读（缓存值或最新值），不会出现空白页面。
> **无缓存时（新用户首次使用）**：数据卡片显示"--"，顶部显示同步状态；同步成功后数字渐入，同步失败或无设备时显示"--"+ 顶部原因文案（见 3.2）。

### 4.3 顶部同步状态条

**同步中样式**：
- 位置：页面最顶部（问候语上方）
- 样式：loading 旋转图标 + "数据同步中..." 文字
- 打开 App 后立即展示，同步完成后淡出消失

**数据过期样式**：
- 样式：圆角条，浅灰色背景，左侧图标 + 原因文案 + 右侧"重试"按钮
- 可关闭：用户可点 × 关闭，下次打开 App 重新判断
- 最多显示一条原因，优先展示用户最可操作的那条
- 重试行为：点击后重新同步所有数据，状态条切回"同步中"。重试仍失败则状态条恢复为原因提示

### 4.4 同步成功过渡动画

| 场景 | 动画效果 |
|:---|:---|
| 数值更新（有旧值） | 数字从旧值滚动到新值（CountUp） |
| 首次出现（无旧值） | 数字渐入 |
| 功能卡片内容更新 | 新旧内容交叉淡入淡出 |

### 4.5 数据过期卡片标注

当全局状态为数据过期时，所有数据卡片统一：
- 数值仍显示上次缓存的值（不留空白）
- 数值下方增加灰色小字显示上次更新时间
- 数值颜色降低透明度，与正常数值形成视觉区分

## 五、触发与刷新逻辑

### 5.1 触发流程

1. 用户打开 App → 顶部显示"数据同步中..."，数据区域展示缓存值
2. 并行查询所有数据类型（睡眠、心率、消耗等）
3. 设备数据全部返回后，触发后端计算（高能时段/入睡窗口）
4. 全部完成 → 顶部状态消失，数据更新到最新值；任一失败 → 顶部显示原因提示
5. 同步成功且有新数据时通知 Smart Digest 重新生成；全部"无变化"时不重新生成

### 5.2 数据过期判定

以下任一情况发生，全局状态即进入"数据过期"：

- **设备数据查询超时**：整体查询 10 秒内未全部返回即视为超时
- **后端计算超时或失败**：后端请求 15 秒内未返回视为超时；返回错误时进入过期状态，网络恢复时自动重试（见 5.3）
- **数据不新鲜**：所有查询返回的数据中，最新一条距当前时间超过 **12 小时**，视为数据过期（12 小时可覆盖"昨晚睡觉到今早打开 App"的正常间隔）

### 5.3 刷新触发方式

| 触发方式 | 说明 |
|:---|:---|
| **自动：打开 App** | 每次 App 回到前台时自动触发（距上次同步 > 5 分钟） |
| **手动：下拉刷新** | 用户在首页下拉，强制重新同步所有数据 |
| **自动：网络恢复** | 网络恢复时自动重试后端计算卡片（高能时段/入睡窗口） |

### 5.4 防抖策略

| 触发方式 | 防抖规则 |
|:---|:---|
| 打开 App（自动） | 距上次同步 < 5 分钟 → 不触发，展示上次结果 |
| 下拉刷新（手动） | 最小间隔 10 秒，不受 5 分钟限制，用户主动下拉时立即触发 |
| 重试按钮（手动） | 最小间隔 10 秒，不受 5 分钟限制，重新同步所有数据 |

## 六、与 Smart Digest 共存规则

| 场景 | 数据同步状态 | Smart Digest |
|:---|:---|:---|
| 正常打开，同步成功 | 更新到最新数据 | 可能触发重新生成 |
| 正常打开，同步失败 | 顶部显示原因 + 数据区域展示缓存 | 展示上次缓存的摘要 |
| 冷却期内再次打开，同步成功 | 更新到最新数据 | 不重新生成（冷却期内） |
| 新用户，无数据 | 顶部显示原因文案，卡片显示"--" | 不展示 |

## 七、边界情况

| # | 场景 | 处理方式 |
|:---:|:---|:---|
| 1 | 同步中用户切到后台 | 暂停 UI 反馈，回到前台时按防抖规则判断是否重新触发 |
| 2 | 同步中用户点击卡片进入详情页 | 详情页使用缓存数据，不阻塞导航。返回首页后更新 |
| 3 | 设备已在后台完成同步 | 打开 App 时数据立即可用，顶部状态条闪现后直接过渡到最新值 |
| 4 | 连续失败（反复打开都无法同步） | 顶部状态条静态显示原因，不重复播放出现动画 |
| 5 | 多个数据来源并存 | 以最近一次写入的来源为准，诊断文案基于该来源 |
| 6 | 首次安装弹出授权对话框 | 授权前显示"--"+ 顶部同步中状态；用户授权后立即查询，拒绝后顶部显示权限提示 |

## 八、体验要求

| 要求 | 说明 |
|:---|:---|
| 即时反馈 | 打开 App 后顶部同步状态立即出现，缓存数据立即可读 |
| 单次查询超时 | 10 秒 |
| 动画流畅 | 数值更新动画、状态条出现动画需保持流畅，无卡顿 |
| 并行同步 | 所有数据类型并行同步，不串行等待 |
| 重试策略 | 失败后不自动重试，由下次打开 App 或手动下拉触发（网络恢复自动重试除外） |

## 九、数据缓存

### 9.1 缓存要求

- 每个数据类型独立缓存数值和上次同步成功时间；同步状态和失败原因为全局单一记录
- 同步成功后自动更新缓存和设备来源信息
- 缓存持久化在本地，随 App 卸载清除
- 不主动过期：即使 30 天前的数据也继续展示，带时间标注

### 9.2 "上次更新"显示逻辑

数据过期时，卡片数值旁显示灰色小字：

| 条件 | 显示格式 | 示例 |
|:---|:---|:---|
| 上次同步在今天 | "HH:mm 更新" | "08:15 更新" |
| 上次同步在昨天 | "昨天 HH:mm" | "昨天 23:30" |
| 上次同步更早 | "N 天前" | "2 天前" |
| 从未成功同步 | 不显示时间，仅显示"--" | "--" |
