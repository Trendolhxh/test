>
> **v1.1 变更**：夜间模式核心指标由单一"入睡准备度（🟢/🟡/🔴 三档）"升级为《入睡准备度》产品设计中的"信号灯 + 罗盘阵列"方案（详见 `sleep readiness` 文档）。首期仅上线"身体"一轴，"环境准备度"待自有硬件（温/光/噪/CO₂ 采集）就绪后再补齐。

## 一、背景与目标

### 1.1 背景

当前首页以数据仪表盘形式平铺睡眠评分、心率、消耗、高能时段、入睡窗口、睡眠分析等信息，存在以下问题：

1. **缺少主线**：信息平铺无层级，用户需要自行综合判断"今天精力好不好"
2. **核心指标弱化**：高能时长作为产品核心概念，在当前首页中仅作为一个普通卡片存在，未建立起用户心智
3. **缺乏行动指引**：用户看完数据后不知道该做什么，产品停留在"告诉你"而非"引导你"
4. **场景未区分**：晨起和睡前是两个截然不同的使用场景，当前首页用同一套布局应对

### 1.2 目标

将首页从"数据仪表盘"改版为**以高能时长为核心、区分状态查看与行动管理的双 Tab 结构**：

| 目标 | 衡量方式 |
|:---|:---|
| 用户 3 秒内获得今日精力结论 | 首页首屏聚焦核心指标变动，不需要滑动 |
| 强化高能时长概念认知 | 高能时长作为首页恒定锚点，每次打开都看到 |
| 提供可执行的行为建议 | 今日 Tab 以时间轴 + 行为 pin 呈现具体建议 |
| 覆盖晨起和睡前两个核心场景 | 状态 Tab 白天/夜间自动切换核心指标 |

### 1.3 设计原则

| 原则 | 说明 |
|:---|:---|
| **结论先行** | 核心指标的变动和好坏判断在首屏第一眼完成，不依赖用户解读 |
| **行为归一** | 所有行动建议统一为"行为 + 时间点 + 剂量"格式 |
| **场景自适应** | 同一套交互结构，通过数据内容切换适应白天/夜间场景 |
| **可视化优先** | 核心信息通过图形和数据可视化传递，文字摘要作为辅助 |
| **拓展性** | 新增归因因子、身体指标、行为类型时，不需要改变首页结构 |

## 二、首页架构：双 Tab 结构

### 2.1 Tab 定义

首页分为两个平级 Tab，用户可左右切换：

| Tab | 名称 | 核心问题 | 定位 |
|:---|:---|:---|:---|
| 状态 | 状态 | "我怎么样？" | 查看精力状态、归因分析、身体数据 |
| 今日 | 今日 | "我该做什么？" | 查看时间轴、行动建议、精力管理 |

### 2.2 默认行为

- **默认 Tab**：状态 Tab（用户 70% 的场景只需要看一眼核心指标）
- **记忆上次 Tab**：不记忆。每次打开 App 始终回到状态 Tab
- **切换方式**：顶部 Tab 栏点击切换，或左右滑动切换

### 2.3 数据同步与 Tab 的关系

数据同步状态层（参见 `data-sync-status-prd.md`）位于 Tab 栏上方，作为全局指示，不随 Tab 切换变化。同步完成后，两个 Tab 的数据同步更新。

```
┌─────────────────────────────────────┐
│  [全局同步状态条]                     │  ← 同步中 / 数据过期时显示
├─────────────────────────────────────┤
│  [ 状态 ]    [ 今日 ]                │  ← Tab 栏
├─────────────────────────────────────┤
│                                     │
│  （当前 Tab 内容区）                  │
│                                     │
└─────────────────────────────────────┘
```

## 三、状态 Tab

### 3.1a 核心指标胶囊（两个 Tab 共用 · v1.3）

核心指标是首页的恒定锚点——**无论用户在状态 Tab 还是今日 Tab，每次打开都要在首屏第一眼看到同一枚"核心指标胶囊"**，不允许两 Tab 出现不同形态。

**胶囊定义**：

| 要素 | 白天模式 | 夜间模式 |
|:---|:---|:---|
| 主数值 | 高能时长（如 `7.5h`） | 入睡准备度状态词（如"接近就绪"）+ 信号灯色 |
| 变化/辅助 | 与昨日变化量（如 `↑ +0.3h`） | 推荐入睡时间（如 `23:00`） |
| 时段/细节 | 高能时段列表（如 `09:30–11:45 · 18:45–21:10`）<br>**仅状态 Tab 胶囊展示；今日 Tab 胶囊省略时段（时段在时间轴底色带上已表达）** | 无 |

**胶囊在两个 Tab 的表现规则**：

| Tab | 位置 | 视觉权重 | 展开差异 |
|:---|:---|:---|:---|
| 状态 Tab | 第一层（见 3.2） | 完整大卡：数值 + 变化 + 时段 + 进度条 | 下方承接 Smart Digest、归因等层级 |
| 今日 Tab | 时间轴顶部恒定锚点，吸顶或随滚动小幅缩略 | 紧凑胶囊：数值 + 变化（**不显示时段**，时段由时间轴色带承担） | 下方直接衔接"今日特色总结"一行 + 时间轴 |

**不允许的表现**（真机走查失分点）：

- ❌ 今日 Tab 顶部把核心指标压成一行指令式横条（"今天高能总长 7.5h，比昨天多 0.3h——重点事放进 09:30–11:45 或 18:45–21:10"），这同时混淆了"核心指标"与"今日特色总结"的层级
- ❌ 两个 Tab 的核心指标字号、颜色、位置不一致——会让用户切 Tab 时重建认知
- ❌ 把"重点事放进 xx–xx"这类行动指令塞进胶囊——行动建议的归属是今日 Tab 时间轴上的 pin 或"今日特色总结"，**不是胶囊**

**"今日特色总结"与胶囊的边界**（详见 `today-tab-prd.md` §7.3）：

- 胶囊 = 核心指标的**当前值与变化**
- 今日特色总结 = 一句**data-driven 的事实判断 + 可行动含义**（如"今天高能比平常多半小时，难事留给你"）
- 二者在今日 Tab 上下分立展示，不合并成一行

### 3.1 整体信息结构

状态 Tab 的信息结构自上而下为五层，形成**核心指标 → 解读 → 归因 → 异常 → 完整数据**的递进关系：

```
┌─────────────────────────────────────┐
│  第一层：核心指标变动                  │  ← 首屏核心，3 秒结论
│  第二层：Smart Digest                │  ← AI 一句话解读 + Agent 入口
│  第三层：数据归因                     │  ← 各因子对核心指标的贡献
│  第四层：值得注意的值                  │  ← 异常数据高亮（仅异常时出现）
│  第五层：完整身体数据                  │  ← 当前数据 + 睡眠数据图表
└─────────────────────────────────────┘
```

用户的典型浏览深度：
- **70% 的场景**：只看第一层（核心指标），确认"今天怎么样"后关闭
- **20% 的场景**：看到第二层（Smart Digest），被一句话吸引后点击进入 Agent 对话
- **10% 的场景**：滑到第三~五层，深入了解归因和完整数据

### 3.2 第一层：核心指标变动

#### 3.2.1 白天模式（默认）

核心指标为**今日高能时长**，以小时数直接展示，配合与昨日的变化量。

**展示要素（首层卡片 · 白名单，v1.3 更新）**：

| 要素 | 说明 | 示例 | 必需/可选 |
|:---|:---|:---|:---:|
| 高能时长数值 | 今日预估/实际高能时长，大字展示 | **7.5h** | 必需 |
| 变化量 | 与昨日的差值，带方向 | **+0.3h** / **-1.2h** | 必需 |
| 高能时段分布 | 具体时间段 | 09:30–11:45 · 18:45–21:10 | 必需 |
| 好坏色彩 | 按下方"可视化规则"表决定卡片主色 | 绿/橙/中性 | 必需 |

**白名单外的元素 UI 不得出现**（真机走查失分点，v1.3 新增红线）：

| 元素 | 为何不要 |
|:---|:---|
| "全部可用 7.5h" 刻度 | 参照系不明（是 10h？24h？可用上限？）；高能时长不是进度条要跑完的终点，用户会把它误读成"今天目标没达成" |
| "👍 状态正常，坚持优化作息冲击 10h+" | PRD 未定义此 CTA；"冲击 10h+" 语气是 gamification，与首页"calm 非激励"调性相反 |
| 时段双次呈现 | 高能时段仅在本卡呈现一次。今日 Tab 胶囊 / 顶部总结条 **不得**再重复列出 `09:30–11:45 · 18:45–21:10` 字符串，时段信息在今日 Tab 由时间轴色带承担 |
| 大数字 `7.5h` 出现 3 次以上 | 状态 Tab 卡片内**只出现一次主数值**。进度条上的刻度标注不得复述主数值 |

> 若未来需要新增 CTA 或辅助元素（如连续达标徽章、个性化目标），必须进本 PRD 白名单并写明触发条件，不得在 UI 实现阶段临时追加。

**好坏判断的可视化规则**：

高能时长不使用抽象评分。通过以下规则让用户直观感知好坏：

| 条件 | 视觉表现 | 说明 |
|:---|:---|:---|
| ≥ 10h | 强正向视觉（如品牌色高亮 + 向上箭头） | 精力充沛 |
| 4h–10h | 常规视觉 | 正常范围 |
| < 4h | 警示视觉（如暖色提示） | 精力不足 |
| 日变化 ≥ +1h | 正向变化强调（如绿色 + 上升标记） | 明显提升 |
| 日变化 ≤ -1h | 负向变化强调（如橙色 + 下降标记） | 明显下降 |
| 日变化 -1h ~ +1h | 平稳视觉，不特别强调 | 正常波动 |

> 设计要求：用户不读数字，仅凭颜色和方向即可判断"今天好不好"。

#### 3.2.2 夜间模式

当最后一个高能时段结束后，核心指标自动从"今日高能时长"切换为**入睡准备度**——对应《入睡准备度》产品设计中"身体 + 环境"两轴架构的身体一轴（见 `sleep readiness` §3）。首期既然只有身体一轴，对用户就叫"入睡准备度"，不暴露"身体/环境"两轴命名以免误以为还有其他维度。

> **首期范围说明**：
> - 首期只上线入睡准备度单轴（内部等价于"身体准备度"）。
> - "环境准备度"（房间温度、光、噪声、CO₂）需要自有硬件采集，**待硬件就绪后再把入睡准备度拆分为"身体 + 环境"两枚信号灯并列**，届时命名同步调整。
> - 湿度、PM2.5 等 T4 因子当前无法直接测量，**UI 上不占位**，避免给用户"缺失感"。

**切换触发条件**：当日所有预测高能时段均已结束（以最后一个高能时段的 `end` 时间为准）。

**布局原则**：夜间模式核心指标区的布局结构与白天"今日高能时长"**尽可能对齐**——相同的标签→主体→附加信息→细颗粒因子的纵向节奏。切换到夜间时：

1. **顶部过渡条**：先显示一条紧凑的"**今日高能 · 已结束**"过渡条（复用能量罐的视觉样式但饱和度降低、填满已消耗），向用户交代"白天的主指标已经收尾"。**不展示 `6.8h` 这类具体小时数**——"已结束 / 已归零" 的语义与具体数值并列会互相矛盾（见 v1.3 走查失分点）。如需展示实际值，请进入状态 Tab 的"完整身体数据"层查看历史。
2. **主体**：下方承接入睡准备度的状态测量。

**展示要素（夜间模式）**：

| 位置 | 要素 | 说明 |
|:---|:---|:---|
| 顶部过渡条 | 今日高能 · 已结束 | 紧凑一行 + 一条灰度能量罐（已完全消耗态），**不展示数字、不展示方向箭头、不展示变化量** |
| 主标签 | "入睡准备度" | 与白天"今日高能时长"位置一致，视觉权重相当 |
| 主体（第一眼） | 3 种状态信号灯 + 推荐入睡时间 | 一眼看清"适不适合入睡"+"推荐几点睡"——**不展示分值**、**不展示 f_S/f_C/f_W 子分**、**不展示明日高能预估** |
| 顶部一行文本（可选） | 暖橙区的一句软化文案 | 例"身体还没进入休息模式，给它一点时间降温"——仅橙区展示 |
| 罗盘阵列（第二眼） | 4 枚偏移罗盘 | 定义见 3.2.3 |

> **为什么不展示分值**：按 `sleep readiness` §1.2 状态语言优先于数值语言、§8.1 orthosomnia 缓解——睡前盯着一个 0–100 的分数会强化"考核"心态，反而延长入睡。3 种状态的直观展现即可回答用户的核心问题（适不适合入睡）。分值在算法内部计算，用于决定信号灯分档和扣分归因，但不在 UI 呈现。

**信号灯 3 种状态**（严格对齐 `sleep readiness` §5.2 的色彩语言）：

| 状态 | 内部档位 | 视觉 | 状态词（示例） | 附加信息 |
|:---|:---|:---|:---|:---|
| 凉白（就绪） | 分值 ≥ 75 | 安静的正向光晕，冷白色调 | "已准备好"、"放松就绪" | 推荐入睡时间；不展示任何建议 |
| 暖黄（接近就绪） | 分值 50–74 | 中性光晕，暖黄色调 | "接近就绪" | 推荐入睡时间；不主动展示建议，用户下钻看偏离因子 |
| 暖橙（准备中） | 分值 < 50 | 轻度提示光晕，**暖橙，不是红色** | "还在准备中" | 推荐入睡时间 + 一句软化文案；具体建议需主动下钻 |

**算法内部仍采用乘法核**（对用户不可见）：

```
内部分值 = 100 × f_S × f_C × f_W
f_S / f_C / f_W 子分仅用于信号灯分档与扣分归因，不在 UI 展示
```

乘法核尊重 Borbély 双过程模型的瓶颈钳制——任一子分过低都会拉低总分，不允许高分项掩盖低分项。子分公式、权重、个人基线（2–4 周后切换为贝叶斯更新）详见 `sleep readiness` §4。

**信号灯视觉与语言红线**：

- **不使用红色**。低分档用暖橙（amber），不是 🔴 红灯；不使用警告图标或惊叹号。
- **不使用 😊😐☹️ 等表情图标**，在健康爱好者人群里显得幼稚。
- **文案软化**："房间稍偏暖"、"HRV 比平时低一些"——不使用"不达标"、"差"、"糟"、"不佳"。
- **临睡前 30 min 内不主动推送**任何负面信息或行动建议（orthosomnia 缓解，见 `sleep readiness` §8.1）；用户主动下钻才展示。
- **凉白和暖黄状态下不主动展示任何即时建议**——睡前场景的体验目标是"安心"，不是"被要求做事"。
- **"隐藏分数"模式**：设置中允许用户关闭数字，只保留信号灯和罗盘颜色/指针。
- **罗盘/信号灯过渡动画**：> 400 ms 缓动，不用急速抖动。

**模式切换的视觉过渡**：
- 切换时不做突兀的页面替换，核心指标区域的内容平滑过渡
- 下方的归因、值得注意的值等层级结构保持不变，内容替换为入睡准备度相关

#### 3.2.3 入睡准备度罗盘阵列（夜间模式专属）

信号灯下方同屏展示 **4 枚偏移罗盘**，让用户一眼看清"是哪个因子在偏离"。罗盘不是进度条——而是在"理想区间"上方漂浮的指针，指针的偏离方向和幅度即是语义（详见 `sleep readiness` §5.3）。

**因子进入首页的四条筛选原则**（全满足）：
1. **即时可测**：当下有实时读数，不是累积/统计量。
2. **睡前可变**：用户能在睡前窗口内通过行动改变它。
3. **语义可视化**：能映射到双向罗盘、单向罗盘或趋势箭头之一。
4. **权重足够**：对当晚入睡准备度的边际贡献 > 5%。

**首期首页 4 枚罗盘**：

| 因子 | 形态 | 含义与视觉规则 |
|:---|:---|:---|
| **HRV** | 双向罗盘（视觉一致）+ 实际仅关心偏低 | 低于个人基线提示交感未退；右侧"高"区标注"放松"保持语言一致 |
| **HR** | 双向罗盘 | 偏离个人基线两侧均扣分；指针-理想区间语义清晰 |
| **远端皮温趋势** | ↗ 趋势箭头 + 状态词 | 关键信号是"是否在上升"（过去 30 min 斜率），而非绝对值；状态词："正在升温 / 平稳 / 未启动" |
| **睡眠压力**（Process S） | **单向罗盘**（产品简化版）| 越高越适合入睡；理想区间落在轨道**右侧（高）**一端；偏低时指针在左侧"仍需累积"区，充足时指针进入右侧"充足"区。**状态词只能用"仍需累积 / 接近充足 / 充足"，禁止与"略浅 / 略深 / 偏浅"等睡眠深度词汇混用**（见 v1.3 走查失分点） |

**关于"睡眠压力"为什么用单向**：

- 学术上 Process S 的映射（清醒时长）是双向的——过低（睡得不够累）和过高（过度疲劳）都不好（见 `sleep readiness` §4.4）。
- 产品上做简化：在用户实际使用时段（睡前窗口内），"过度疲劳"是少数边缘情况，而"累积不够"是主流扣分原因。把睡眠压力单向化（越高越好），配合"过度疲劳"单独走下钻归因的长期洞察，用户心智更简洁、首页罗盘语言更统一。
- 对"过度疲劳"（清醒时长过长）的保护由算法侧保留：内部 f_S 仍是双向打分，在极端高位时会扣分并在归因层以"你已经超过 18 小时没合眼了，强行推迟反而更难入睡"的语言浮现，但不在首页罗盘上反映。

**罗盘五要素**（适用于双向/单向罗盘）：
1. 理想区间（浅色高亮带；双向居中，单向靠一侧）
2. 当前位置指针（小圆点或三角）
3. 当前数值 + 单位
4. 偏离方向色调（微妙，强度低于信号灯，仅作辅助）
5. 上下文状态词（见下表 · v1.3 规范）

**状态词规范表（v1.3）**——不同因子使用独立的状态词集合，**禁止跨因子复用**，避免出现"睡眠压力 13.5h 略浅"这类语义错位：

| 因子 | 允许的状态词 | 禁用词 |
|:---|:---|:---|
| HRV（双向） | "稍低 / 略低 / 平稳 / 略高 / 偏高" | — |
| HR（双向） | "稍低 / 略低 / 平稳 / 略高 / 偏高" | — |
| 远端皮温趋势 | "正在升温 / 平稳 / 未启动" | — |
| 睡眠压力（单向，higher-better） | "**仍需累积 / 接近充足 / 充足**" | ❌ "略浅 / 略深 / 偏浅"（与睡眠深度混淆）<br>❌ "不足 / 偏低"（负向措辞） |

**单位呈现规则（v1.3）**：

- HRV、HR 的单位（`ms`、`bpm`）是用户熟悉量纲，保留展示
- 睡眠压力的内部字段单位是 `h`（清醒时长），但 UI 侧**可选择不展示单位**——避免用户把"13.5h"误读为"睡了 13.5 小时"或"睡眠质量 13.5 分"。若展示，应加前缀如"已累积 13.5h 清醒时间"，不得裸写"13.5h"
- 远端皮温趋势不展示绝对数值，仅展示方向箭头 + 状态词

**不上首页的因子**（移至下钻归因层）：

| 因子 | 不上首页原因 |
|:---|:---|
| 睡前体动 | 夜间用户基本都在静止状态，二值信号长期没变化、信息量低（首页视觉预算更值得留给会变化的因子）——移至下钻归因层 |
| 咖啡因残留、晚餐、酒精、尼古丁 | T3 行为因子，已通过 HR/HRV 回传；再独立呈现会重复计数 |
| 运动负荷 | 累积量，不适合"当下"判断 |
| 相位契合（昼夜节律偏早/偏晚） | 抽象推导量，放在下钻的"可能原因"里更合适 |
| 呼吸率 | 信息量与 HR/HRV 高度相关，避免首屏信号过载 |
| 环境温度 / 光 / 噪声 / CO₂ | 依赖自有硬件，首期暂不上线 |

> **信号密度红线**：首页最多呈现 7±2 个信号元素（信号灯 + 推荐入睡时间 + 4 枚罗盘 + 可选一行文本 = 6–7）。超出此上限的因子全部进入下钻层。

### 3.3 第二层：Smart Digest

复用现有 Smart Digest 能力（参见 `homepage_smart_digest.md`），位置调整为**核心指标下方**。

**展示形式**：
- 一行 AI 生成的文字（≤ 50 字），对核心指标变动的人话解读
- 整行可点击，点击后展开进入 Agent Chatflow 对话

**与现有 Smart Digest PRD 的差异**：

| 维度 | 原方案 | 改版后 |
|:---|:---|:---|
| 位置 | 问候语下方副标题 | 核心指标下方 |
| 触发条件 | 不变（app_open + data_refreshed + 冷却期 + 数据有变化） | 不变 |
| 内容生成 | 不变（基于 24h 数据 diff 生成） | 不变 |
| 交互 | 快捷提问按钮点击进入 Chatflow | Smart Digest 文字本身可点击展开进入 Chatflow；快捷提问按钮移至第四层"值得注意的值"附近 |

**Smart Digest 在夜间模式下的适配**：
- 夜间模式下，Smart Digest 的生成输入包含入睡准备度（内部 f_S / f_C / f_W 子分及归因证据，子分不在 UI 展示但可进入 digest 生成 prompt），摘要内容偏向睡眠准备方向
- 示例（安全）："静息心率已经降到 58 了，身体比你先准备好了"

**临睡前 30 min 红线（v1.3 强化）**：

- **窗口定义**：`|now − recommended_bedtime| ≤ 30 min`，生成端必须拿到 `recommended_bedtime` 做判定
- **在窗口内 Smart Digest 必须满足**：
  - 语气中性或安抚；**不出现任何含具体时长/分钟量级的行动建议词**（"再等 X 分钟"、"X 分钟后入睡"、"做 X 分钟呼吸"等都禁止）
  - 不出现"你还没准备好 / 差 / 不达标 / 晚了 / 最好赶紧" 等负面或催促词汇
  - 不出现需要用户立刻执行的 CTA；如果算法判断确有必要推建议，**建议走下钻层**（可能原因 / 可能原因下的 Agent 对话），而非顶部摘要
- **DO / DON'T 示例**：

| 场景 | ❌ DON'T（真机走查违规样例） | ✅ DO |
|:---|:---|:---|
| 22:30，推荐入睡 23:00（窗口内） | "心率还在慢慢降，距上次运动才 2 小时——**再等 25 分钟**会更容易入睡" | "心率在慢慢降，身体在往放松方向走" |
| 23:15，推荐入睡 23:00（窗口内，已略超时） | "你超过推荐入睡 15 分钟了，抓紧睡" | "此刻身体已经接近休息模式" |
| 22:00，推荐入睡 23:00（窗口外） | （正常建议式措辞允许，但仍避免红色/惊叹号） | "离推荐入睡还有一小时，可以先做放松准备" |

- 生成端实现建议：30 min 窗口判定通过后，把 prompt 中的"建议"类输出槽位关闭，仅保留"状态描述"槽位

### 3.4 第三层："为什么是这样"

归因层解释核心指标"为什么是这个值"，将影响因子拆解展示。

**UI 标题统一（v1.3）**：白天模式和夜间模式的本层入口在用户可见标题上统一为 **"为什么是这样 >"**，**不再使用"数据归因"、"可能原因"、"归因分析" 等工程/内部术语**。"可能原因（最多 3 条）" 这类把内部规则写进标题的写法同步废弃（见 3.4.2）。文档正文仍保留"归因"作为内部术语。

#### 3.4.1 白天模式归因

归因对象：**今日高能时长**

入口 UI 文案：`为什么是这样 >`

**归因因子**：

| 因子 | 数据来源 | 贡献方向示例 |
|:---|:---|:---|
| 昨夜睡眠时长 | Apple Health 睡眠数据 | 睡了 7h12m → +0.2h |
| 深睡占比 | Apple Health 睡眠数据 | 深睡 22%（↑3%）→ +0.1h |
| 昨日运动 | Apple Health 活动数据 | 跑步 30min → +0.2h |
| 咖啡因摄入 | 用户记录 / 自动检测 | 10:00 1 杯 → 中性 |
| 入睡时间偏差 | Apple Health 睡眠数据 | 比推荐晚 45min → -0.3h |

**展示规则**：
- 每个因子一行：因子名称 + 具体数据 + 对高能时长的贡献方向和幅度
- 贡献为正用正向视觉标记，贡献为负用负向视觉标记，中性不标记
- 因子按贡献绝对值排序，影响最大的排最前
- 可视化形式建议：紧凑的横向条形图或瀑布图，让用户直观看到各因子的正负贡献

**拓展性**：未来新增归因因子（光照时长、CO2 浓度、药物等），只需在此列表中增加一行，不影响整体结构。

#### 3.4.2 夜间模式归因

归因对象：**入睡准备度**

入口 UI 文案：`为什么是这样 >`（与白天一致）。**"最多 3 条"是内部规则，不得写进入口标题**（历史样例"可能原因（最多 3 条）"在 v1.3 废弃）。

夜间归因层在语义上是 `sleep readiness` §6.2 所说的"可能原因"——**这是行为因子（T3）唯一的第一人称出场位置**，也是内部 f_S/f_C/f_W 子分唯一在 UI 上以文字形式暴露的地方（以"哪一类准备不足"的方式出现，不展示具体分值）。

**证据来源分层**：

| 证据层 | 获取方式 | 示例 |
|:---|:---|:---|
| **直接测量的生理异常（T1）** | T1 信号当前值 vs 个人基线 | "HRV 比过去 14 天基线低 28%" |
| **本日行为记录回放（T3）** | T3 用户自报 + 时间戳 + 药代/生理模型 | "15:30 摄入 200 mg 咖啡因，预计当前残留 110 mg" |
| **被动推断的行为（T2）** | 算法识别（IMU、HR 组合推断） | "检测到 19:00–19:30 有中高强度运动，可能尚未完全恢复" |
| **长期模式** | 个人基线对比 | "你最近 7 晚入睡时间都晚于习惯窗口 > 45 分钟" |
| ~~环境累积暴露~~ | ~~T1 环境传感器时序积分~~ | _待自有硬件就绪后上线（例："过去 2h 平均光照 320 lux"）_ |

**展示规则**：

- 归因条目按**对当晚入睡准备度内部分值的扣分贡献绝对值降序**排列
- **最多展示 3 条**——超过 3 条会触发信息过载 / orthosomnia
- 每条包含：证据短句 + "准备类别"小标签（睡眠压力 / 昼夜节律 / 唤醒抑制，对应 f_S / f_C / f_W 的产品化表述）+ 扣分贡献幅度（可视化条/数字，不用红色）
- **已不可改变的因子**（如已过去的晨光缺失）仍可呈现，但标注"这条无法当晚改变，建议移到长期计划"
- **无归因条目时**（凉白区、各因子都在理想区间）：不展示本层，页面自然收缩

**夜间归因不展示行动 CTA**。行动建议走"即时建议"路径（见第七节），每晚最多 2 条，且临睡前 30 min 内不主动推送。

### 3.5 第四层：值得注意的值

当某个身体指标出现异常（特别好或特别坏）时，以卡片形式凸显。

**触发规则**：

| 异常类型 | 判定条件示例 | 展示方式 |
|:---|:---|:---|
| 正向异常 | 深睡占比 28%，超过近 30 天 P90 | 正向卡片（品牌色边框），标题 + 一句说明 |
| 负向异常 | 静息心率 +8bpm，超过近 7 天均值 2 个标准差 | 警示卡片（暖色边框），标题 + 一句说明 |
| 高能时长异常波动 | 日变化 ≥ ±1h | 在第一层核心指标处已有视觉强调，此处不重复 |

**交互**：
- 每个异常卡片可点击 → 进入 Agent Chatflow，自动注入该异常数据作为对话上下文
- 注入方式同 Smart Digest：异常描述作为 assistant 首条消息，用户的点击等同于一条预设提问（如"静息心率为什么偏高"）

**无异常时**：第四层不显示，页面自然收缩，第三层直接连接第五层。

**Smart Digest 快捷提问**：3 个快捷提问按钮放置在本层附近（有异常时在异常卡片下方，无异常时在第三层下方），作为进入 Chatflow 的低摩擦入口。

### 3.6 第五层：完整身体数据

分为两个子区块，提供完整的身体数据查看：

#### 3.6.1 当前身体数据

| 指标 | 可视化形式 | 说明 |
|:---|:---|:---|
| 心率 | 实时数值 + 日内趋势折线图 | 静息心率高亮标注 |
| HRV | 数值 + 近 7 天趋势折线图 | 标注健康参考区间 |
| 活动量 | 今日步数 + 活动消耗 | 环形进度或条形 |
| 血氧 | 平均值 + 最低值 | 异常低值时高亮 |

#### 3.6.2 睡眠数据

| 指标 | 可视化形式 | 说明 |
|:---|:---|:---|
| 睡眠时长 | 数值 + 近 7 天趋势柱形图 | 标注推荐时长区间 |
| 睡眠阶段 | 水平堆叠条形图（深睡/浅睡/REM/清醒） | 各阶段占比和时长 |
| 入睡/起床时间 | 数值 + 近 7 天一致性散点图 | 偏差过大时标注 |
| 睡眠效率 | 百分比 + 近 7 天趋势 | 标注健康参考区间 |

> 此区块为深度查看需求服务，大多数用户不会每天翻到这里。数据均来自 Apple Health 同步结果。

### 3.7 夜间模式切换

#### 3.7.1 切换时机

- **触发条件**：当日所有预测高能时段的最后一个 `end` 时间已过
- **示例**：高能时段为 09:30–11:45 和 18:45–21:10，则 21:10 后自动切换为夜间模式
- **反向切换**：次日 05:00 后首次打开 App 自动恢复为白天模式（参照日期分界线规则，见 `sleep-window-prd.md` 2.1.1）

#### 3.7.2 切换内容对照

| 层级 | 白天模式 | 夜间模式 |
|:---|:---|:---|
| 第一层 | 今日高能时长 + 变化量 | "今日高能 · 已结束"过渡条（**不含具体数值**）+ 入睡准备度（3 状态信号灯 + 推荐入睡时间）+ 罗盘阵列（HRV / HR / 皮温 / 睡眠压力）。**不展示**分值、子分、明日高能预估 |
| 第二层 | Smart Digest（高能 / 睡眠相关） | Smart Digest（入睡准备相关，临睡前 30 min 内避免负面措辞） |
| 第三层 | 入口标题"**为什么是这样 >**"；内容为高能时长归因因子（瀑布图，全量呈现） | 入口标题"**为什么是这样 >**"；内容为入睡准备度归因证据（**最多 3 条作为内部规则，不写进标题**，扣分贡献降序） |
| 第四层 | 身体指标异常 | 入睡相关指标异常（使用软化文案） |
| 第五层 | 不变 | 不变 |

> 交互结构完全一致，用户学一次就会用，夜间模式不引入新的交互模式；视觉语言上保持"平静 (calm) 而非激励 (gamified)"的基调（见 `sleep readiness` §1.2）。

### 3.8 重要日场景（2b）

**场景描述**：用户明天有重要事，想确保明日精力最好。

**入口**：状态 Tab 进入夜间模式后，Smart Digest 或快捷提问中出现相关引导（如"明天有重要的事怎么准备"）。

**用户流程**：

```
状态 Tab（夜间模式）
    ↓ 用户点击 Smart Digest 或快捷提问
    ↓
进入 Agent Chatflow
    ↓ Agent 基于当前入睡准备度（内部 f_S/f_C/f_W 子分）和归因证据
    ↓ 生成个性化的行动建议方案
    ↓ 建议格式：行为 + 时间点 + 剂量
    ↓ 例："22:00 做 10 分钟呼吸练习 → 预计将唤醒抑制从'偏低'拉回'充足'"（Agent 对话里可以提及内部分值变化，但首页 UI 不展示）
    ↓
Agent 展示"一键添加到今日"按钮
    ↓ 用户点击
    ↓
行动建议作为 pin 添加到今日 Tab 的时间轴
```

**Agent 建议生成约束**（对齐 `sleep readiness` §7.1–§7.2）：

- 建议按"对当晚分数的扣分贡献"降序生成，排除已不可改变的因子（例：咖啡因已摄入则移到长期主题，不放入今晚即时建议）
- 即时建议**最多 2 条**——多条并列会触发 orthosomnia，反而延长入睡
- 预期生理效应加权：开窗降 CO₂（分钟级）> 呼吸练习（10 min）> 穿袜助 DPG（20–30 min）
- 极简 CTA 文案："关主灯"、"4-7-8 呼吸 5 分钟"——避免说教

**体验目标**：专业指导 + 掌控感。Agent 提供因果透明的建议（做什么 → 改善多少），用户选择后能一键添加到行动计划中。

## 四、今日 Tab

### 4.1 整体信息结构

今日 Tab 是用户的**精力管理工具**，以**纵向 24 小时时间轴**为核心，时间从上往下流动，将睡眠时段、能量区间、行动建议统一展示在一条可滚动的垂直时间线上。

```
┌──────────────────────────────────────────────┐
```

### 4.2 纵向 24h 能量时间轴

#### 4.2.1 核心定义

一条从上到下覆盖 24 小时的**纵向可滚动时间轴**，是今日 Tab 的视觉主体。用户通过上下滑动浏览全天。

**布局结构**：

| 要素 | 说明 |
|:---|:---|
| 时间轴方向 | **纵向**，时间从上到下流动 |
| 起点 | 用户常规起床时间前 1h（如常规 06:30 起床，从 05:30 开始） |
| 终点 | 次日同一时间（覆盖完整 24h + 入睡时段） |
| 左侧时间标签 | 每小时一个标签（05、06、07…00、01），固定在时间轴左侧边缘 |
| 中央区域 | **连续时间线**：无区块分割，以能量曲线 + 背景色带展示全天能量变化 |
| 右侧 | **行动 pin**：以浮动标签形式钉在对应时间位置 |
| 能量曲线 | 贯穿全天的纵向贝塞尔曲线（X=能量水平，Y=时间），向右=高能，向左=低能 |

#### 4.2.2 视觉分层

时间轴采用**连续时间线 + 背景色带高亮**的方式，不划分独立区块：

| 视觉层 | 说明 |
|:---|:---|
| **时间刻度** | 每小时一条水平参考线 + 左侧时间标签 |
| **睡眠区底色** | 紫色半透明底色覆盖昨夜睡眠时段（0:00 → 起床时间），叠加睡眠时长和效率文字 |
| **高能时段底色** | **品牌绿色半透明底色**覆盖高能窗口区间，左上角标注 ⚡ 高能 + 起止时间 |
| **能量曲线** | 一条贯穿全天的纵向贝塞尔曲线，展示能量水平的连续变化（详见 4.2.4） |
| **入睡窗口** | 紫色半透明底色 + 🌙 入睡窗口卡片（推荐入睡时间 + 明日预估高能） |
| **行动 pin** | 浮动标签钉在右侧对应时间位置（详见 4.3） |

用户在此页面关注三件事：**高能时段落在什么时间**、**入睡窗口落在什么时间**、**具体的行动建议**。

#### 4.2.3 时间轴的三种状态

**预测态**（晨起，当日首次打开）：

| 内容 | 说明 |
|:---|:---|
| 时间轴 | 全天数据均为预测，统一样式展示 |
| 能量曲线 | 全天曲线统一为虚线（`stroke-dasharray`），透明度降低 |
| 行动 pin | 所有当日建议 pin 可见，状态为"待执行" |
| 自动定位 | 滚动到起床时间附近 |

**实时进度态**（日间打开）：

| 内容 | 说明 |
|:---|:---|
| 当前时间标记 | 一条横线 + 红色圆点 + "现在" 标签，贯穿时间轴宽度 |
| 上方（已过） | 曲线以实线展示（实际数据） |
| 下方（未来） | 曲线以虚线展示（预测数据），通过 SVG clipPath 分割 |
| 行动 pin | 已过的标记完成/跳过，未来的正常展示 |
| 自动定位 | 滚动到当前时间标记附近 |

**回顾态**（睡前/夜间打开）：

| 内容 | 说明 |
|:---|:---|
| 时间轴 | 全天均为实际数据 |
| 能量曲线 | 全天实线展示 |
| 行动 pin | 全部标记为已完成/已跳过 |
| 入睡窗口 | 底部入睡窗口卡片突出展示 |
| 自动定位 | 滚动到晚间入睡窗口附近 |

#### 4.2.4 全天连续能量曲线

一条**贯穿全天**的纵向能量曲线，覆盖从 0 时到入睡窗口后 1h 的完整时间范围（不再仅限于高能区块内部）：

- 使用 Catmull-Rom 样条插值 → 三次贝塞尔 SVG 路径，不使用折线
- Y 轴 = 时间（从上到下流动），X 轴 = 能量水平（向右 = 高能）
- 曲线左侧（曲线 → 左边界）以品牌绿色半透明填充，形成面积图效果
- 睡眠时段能量值极低（曲线贴近左边界），起床后逐渐上升，高能窗口时达到峰值
- 实时态下：当前时间线以上为实线（实际数据），以下为虚线（预测数据），通过 SVG clipPath 实现

#### 4.2.5 可视化要求

| 要求 | 说明 |
|:---|:---|
| 曲线平滑 | 使用贝塞尔曲线或样条插值，不使用折线 |
| 滚动交互 | 支持**垂直滑动**浏览全天；自动定位到当前时间/关键区域 |
| 每小时高度 | 约 80px，整个 24h 时间轴总高度约 1920px，需要滚动查看 |
| 性能 | 区块和曲线渲染 ≤ 1 帧（16ms），滚动流畅 60fps |
| 响应式 | 适配屏幕宽度，时间轴占满 Tab 内容区宽度 |

### 4.3 行动建议事件

#### 4.3.1 事件格式

所有行动建议统一归一化为三要素：

| 要素 | 说明 | 示例 |
|:---|:---|:---|
| **行为** | 建议的具体行为，配合图标 | ☕ 喝咖啡、🏃 运动、😴 小睡、🧘 冥想、🌙 准备入睡 |
| **时间点** | 建议执行的时间或时间窗口 | 10:30 前、17:00–18:00、22:30 |
| **剂量** | 建议的量或时长 | 1 杯、30min、20min |

#### 4.3.2 Pin 在时间轴上的展示

- 每个行动建议以**浮动标签**的形式钉在时间轴**右侧**对应时间位置
- Pin 的视觉形式：圆角小标签，内含行为类型图标（☕🏃😴🧘🌙）+ 简短标签（如"喝咖啡"）
- Pin 的纵向位置对应其建议时间，自然嵌入时间线视觉流中——咖啡 pin 在高能区间前，用户直观理解"为什么是这个时间"
- 多个 pin 时间接近时，自动向下排布避免重叠

#### 4.3.3 Pin 的状态

| 状态 | 视觉表现 | 说明 |
|:---|:---|:---|
| 待执行 | 正常颜色 + 实心图标 | 时间点未到 |
| 当前建议 | 高亮 + 脉动效果（可选） | 距最近的下一个建议 |
| 已完成 | 浅色 + ✓ 标记 | 用户已执行（自动检测或手动标记） |
| 已跳过 | 浅色 + 删除线 | 时间已过但未执行 |

#### 4.3.4 Pin 的交互

点击 pin 展开详情浮层（在时间轴上方或下方弹出）：

| 详情内容 | 说明 |
|:---|:---|
| 行为 + 时间 + 剂量 | 完整信息 |
| 原因说明 | 一句话解释（如"10:30 前摄入咖啡因不影响晚间睡眠"） |
| 替代方案 | 可选，如"改为绿茶（咖啡因更低）" |
| 对高能的预估影响 | 可选，如"预计贡献 +0.3h 高能" |
| 进入 Chatflow | "了解更多"按钮 → 进入 Agent 对话 |

#### 4.3.5 建议来源与完成判定

**建议来源**：

| 来源 | 说明 |
|:---|:---|
| 算法自动生成 | 基于用户睡眠数据、生理节律、历史行为模式自动生成当日建议 |
| Agent 对话生成 | 2b（重要日）场景中 Agent 生成的建议，用户一键添加 |
| 用户手动添加 | 未来支持用户自己添加事件 |

**完成判定**：

| 行为类型 | 判定方式 |
|:---|:---|
| 运动 | Apple Health 检测到运动记录 → 自动标记完成 |
| 小睡 | Apple Health 检测到睡眠记录（白天） → 自动标记完成 |
| 咖啡因 | V1 不自动检测，时间过后标记为"已跳过"；用户可手动标记完成 |
| 冥想 | V1 不自动检测，时间过后标记为"已跳过"；用户可手动标记完成 |
| 准备入睡 | 不标记状态，作为时间提示存在 |

> 未来自有硬件和更多数据源接入后，可提升自动检测覆盖率。

### 4.4 入睡窗口集成

#### 4.4.1 在时间轴上的展示

入睡窗口在时间轴末端以**紫色底色带 + 浮动卡片**展示：

- 位置：时间轴上推荐入睡时间对应位置（底部区域）
- 视觉：紫色半透明背景带 + 🌙 图标 + 入睡窗口卡片（推荐入睡时间 + 明日高能预估）
- 同时在建议列表中有一个 🌙 "准备入睡" 的 pin 标签
- 随一天推进，用户滚动到晚间时入睡窗口自然进入视野

#### 4.4.2 点击进入设置页

点击入睡窗口标记 → 进入睡眠窗口设置页（复用现有 `sleep-window-prd.md` 定义的独立页面）。

设置页功能不变：
- 圆盘时钟调整入睡/起床时间
- 实时预览明日高能时长变化
- 设置入睡提醒和起床闹钟

#### 4.4.3 与状态 Tab 的联动

用户在入睡窗口设置页调整时间后：
- 状态 Tab 夜间模式的"推荐入睡时间"和"明日高能预估"同步更新
- 今日 Tab 时间轴上的入睡窗口标记位置和预估数值同步更新

### 4.5 熬夜场景（2c）

**场景描述**：用户今晚需要晚睡，想知道不同入睡时间对明日精力的影响，选择一个损失最小的方案。

**用户流程**：

```
今日 Tab → 点击时间轴上的入睡窗口标记
    ↓
进入睡眠窗口设置页
    ↓ 用户拖动圆盘将入睡时间从 22:30 调到 01:00
    ↓ 实时预览：明日高能从 7.5h 降到 5.8h（-1.7h）
    ↓ 继续尝试 02:00 → 明日高能 4.5h（-3.0h）
    ↓ 回到 01:00（性价比最高）
    ↓
确认今晚入睡窗口为 01:00
    ↓
返回今日 Tab
    ↓ 时间轴上入睡窗口位置更新到 01:00
    ↓ 如果入睡提醒已开启，提醒时间同步更新
```

**体验目标**：务实信任。不说教"你不应该熬夜"，直接展示不同选择的代价（明日高能变化），让用户自己做最优判断。

**设置页改版要求**：

现有设置页（`sleep-window-prd.md`）已支持拖动调整和实时预览，可完整复用。唯一需要确认的是：
- 当用户选择的入睡时间远晚于推荐时间（如晚 2h 以上），文案上保持"中性呈现差值"原则，不增加负面措辞或警告

### 4.6 未来拓展

今日 Tab 的时间轴 + 行动 pin 架构具备良好的拓展性：

| 拓展方向 | 时间线 | 说明 |
|:---|:---|:---|
| **日历同步** | V2 | 用户授权日历后，日程事件显示在时间轴上。系统根据日程自动调整行为建议（如"下午有重要会议，建议 14:00 喝咖啡"） |
| **事件自动检测** | V2 | 运动结束、小睡结束等事件被自动检测后，在时间轴上生成"已完成"的事件 pin，配合高能时长的实时更新形成正反馈 |
| **手动记录** | V2 | 用户可在时间轴上手动添加事件（喝了咖啡、吃了药、喝了酒），系统将其纳入后续建议的调整因素 |
| **药物影响** | V3 | 药物摄入事件可关联药物数据库，预估对能量曲线的影响，在时间轴上可视化展示 |
| **环境数据** | V3 | 自有硬件提供光照、CO2 数据后，环境因素作为新的 pin 类型出现（如"现在开窗通风 15min → CO2 ↓"） |

## 六、场景体验目标

以下体验目标作为设计评审的验收标准。每个场景的设计方案必须满足"核心体验"和"调性"要求，不得出现"反面"列所描述的感受。

| 场景 | 触发条件 | 核心体验 | 调性 | 反面（不应有的感受） |
|:---|:---|:---|:---|:---|
| **晨起** | 当日首次打开 + 05:00–12:00 | 掌控感 + 效率 | 清晰、果断、轻快 | 信息过载、需要花时间理解 |
| **日间查看** | 非首次打开 / 12:00 后 | 效率 + 即时行动力 | 即时、具体、一步到位 | 信息过多、找不到重点 |
| **睡前常规** | 夜间模式激活后 | 安心（calm，非 gamified） | 安静、温暖、确认；不用红色、不用表情符号、不用"差/糟/不佳"字眼 | 焦虑、被考核、被要求做事、orthosomnia（睡眠表现焦虑） |
| **重要日前夜** | 用户主动触发（Chatflow） | 专业指导 + 掌控 | 专业、可控、即时反馈；建议最多 2 条 | 无力感、选择困难、建议铺天盖地 |
| **需要熬夜** | 用户调整入睡窗口 | 务实信任 | 不评判、策略感 | 被说教、内疚感 |
| **活动结束** | 检测到运动/小睡/冥想结束（V2） | 成就感 + 正反馈 | 仪式感、数字变化 | 无感、被淹没 |

> **睡前调性红线**（对齐 `sleep readiness` §1.2、§5.6、§8.1）：
> - 不用红色、不用徽章/连胜/排行榜、不用 😊😐☹️ 表情图标
> - 状态语言优先于数值语言（"稍偏暖 / 静止"而不是"62 分 / 43 分"）
> - 临睡前 30 min 内不主动推送任何负面信息
> - 提供"隐藏分数"模式，允许用户只看信号灯和罗盘
> - 若识别到用户为失眠困扰人群（onboarding 自标或行为信号），自动切换到"隐藏分数"模式并引导至 CBT-I 功能线，不走准备度主体验

- 引导完成/跳过状态本地持久化，不再重复触发
