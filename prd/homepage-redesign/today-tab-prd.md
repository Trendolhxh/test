## 一、背景与定位

### 1.1 在首页双 Tab 中的位置

首页分为两个平级 Tab：

| Tab | 名称 | 核心问题 | 定位 |
|:---|:---|:---|:---|
| 状态 | 状态 | "我怎么样？" | 查看精力状态、归因分析、身体数据 |
| **今日** | 今日 | **"我该做什么？"** | **查看时间轴、行动建议、精力管理** |

- 默认 Tab 为状态 Tab；今日 Tab 是用户主动切换进入
- 切换方式：顶部 Tab 栏点击或左右滑动
- 每次打开 App 始终回到状态 Tab，不记忆上次 Tab

### 1.2 今日 Tab 的产品定位

**今日 Tab 是用户的精力管理工具**，承担以下核心职责：

| 职责 | 说明 |
|:---|:---|
| **回答"我今天该做什么"** | 以时间轴方式呈现建议，用户知道"什么时候做什么" |
| **连接状态与行动** | 状态 Tab 告诉你"怎么样"，今日 Tab 告诉你"该怎么做" |
| **承接 Agent 输出** | 2b 场景中 Chatflow 生成的行动建议一键落到时间轴上 |
| **精力管理看板** | 展示全天能量曲线、高能时段、睡眠段，形成"一天视图" |

### 1.3 设计原则（今日 Tab 特有）

| 原则 | 说明 |
|:---|:---|
| **时间即主轴** | 纵向 24h 时间轴是信息组织的核心，所有要素围绕时间锚点 |
| **工具型页面** | 纯工具属性，不展示 AI 文字摘要（Smart Digest 仅在状态 Tab） |
| **行为归一** | 所有行动建议统一为"行为 + 时间点 + 剂量"三要素 |
| **拓展友好** | 新增建议类型、事件类型、手动事件时不改结构，只加 pin 类型 |

## 三、纵向 24h 能量时间轴

### 3.1 核心定义

一条从上到下覆盖 24 小时的**纵向可滚动时间轴**，是今日 Tab 的视觉主体。用户通过上下滑动浏览全天。

**布局结构**：

| 要素 | 说明 |
|:---|:---|
| 时间轴方向 | **纵向**，时间从上到下流动 |
| 起点 | 用户常规起床时间前 1h（如常规 06:30 起床，从 05:30 开始） |
| 终点 | 次日同一时间（覆盖完整 24h + 入睡时段） |
| 左侧时间标签 | 每小时一个标签（05、06、07…00、01），固定在时间轴左侧边缘 |
| 中央区域 | **连续时间线**：无区块分割，以能量曲线 + 背景色带展示全天能量变化 |
| 右侧 | **行动 pin**：以浮动标签形式钉在对应时间位置 |
| 能量曲线 | 贯穿全天的纵向贝塞尔曲线（X=能量水平，Y=时间），向右=高能，向左=低能 |

### 3.2 视觉分层

时间轴采用**连续时间线 + 背景色带高亮**的方式，不划分独立区块：

| 视觉层 | 说明 |
|:---|:---|
| **时间刻度** | 每小时一条水平参考线 + 左侧时间标签 |
| **睡眠区底色** | 紫色半透明底色覆盖昨夜睡眠时段（0:00 → 起床时间），叠加睡眠时长和效率文字 |
| **高能时段底色** | **品牌绿色半透明底色**覆盖高能窗口区间，左上角标注 ⚡ 高能 + 起止时间 |
| **能量曲线** | 一条贯穿全天的纵向贝塞尔曲线，展示能量水平的连续变化（详见 3.4） |
| **入睡窗口** | 紫色半透明底色 + 🌙 入睡窗口卡片（推荐入睡时间 + 明日预估高能） |
| **行动 pin** | 浮动标签钉在右侧对应时间位置（详见 §四） |

### 3.3 时间轴的三种状态

**预测态**（晨起，当日首次打开）：

| 内容 | 说明 |
|:---|:---|
| 时间轴 | 全天数据均为预测，统一样式展示 |
| 能量曲线 | 全天曲线统一为虚线（`stroke-dasharray`），透明度降低 |
| 行动 pin | 所有当日建议 pin 可见，状态为"待执行" |
| 自动定位 | 滚动到起床时间附近 |

**实时进度态**（日间打开）：

| 内容 | 说明 |
|:---|:---|
| 当前时间标记 | 一条横线 + 红色圆点 + "现在" 标签，贯穿时间轴宽度 |
| 上方（已过） | 曲线以实线展示（实际数据） |
| 下方（未来） | 曲线以虚线展示（预测数据），通过 SVG clipPath 分割 |
| 行动 pin | 已过的标记完成/跳过，未来的正常展示 |
| 自动定位 | 滚动到当前时间标记附近 |

**回顾态**（睡前/夜间打开）：

| 内容 | 说明 |
|:---|:---|
| 时间轴 | 全天均为实际数据 |
| 能量曲线 | 全天实线展示 |
| 行动 pin | 全部标记为已完成/已跳过 |
| 入睡窗口 | 底部入睡窗口卡片突出展示 |
| 自动定位 | 滚动到晚间入睡窗口附近 |

### 3.4 全天连续能量曲线

一条**贯穿全天**的纵向能量曲线，覆盖从 0 时到入睡窗口后 1h 的完整时间范围（不再仅限于高能区块内部）：

- 使用 Catmull-Rom 样条插值 → 三次贝塞尔 SVG 路径，不使用折线
- Y 轴 = 时间（从上到下流动），X 轴 = 能量水平（向右 = 高能）
- 曲线左侧（曲线 → 左边界）以品牌绿色半透明填充，形成面积图效果
- 睡眠时段能量值极低（曲线贴近左边界），起床后逐渐上升，高能窗口时达到峰值
- 实时态下：当前时间线以上为实线（实际数据），以下为虚线（预测数据），通过 SVG clipPath 实现

### 3.5 可视化要求

| 要求 | 说明 |
|:---|:---|
| 曲线平滑 | 使用贝塞尔曲线或样条插值，不使用折线 |
| 滚动交互 | 支持**垂直滑动**浏览全天；自动定位到当前时间/关键区域 |
| 每小时高度 | 约 80px，整个 24h 时间轴总高度约 1920px，需要滚动查看 |
| 性能 | 区块和曲线渲染 ≤ 1 帧（16ms），滚动流畅 60fps |
| 响应式 | 适配屏幕宽度，时间轴占满 Tab 内容区宽度 |

## 四、行动建议事件

### 4.1 事件格式

所有行动建议统一归一化为三要素：

| 要素 | 说明 | 示例 |
|:---|:---|:---|
| **行为** | 建议的具体行为，配合图标 | ☕ 喝咖啡、🏃 运动、😴 小睡、🧘 冥想、🌙 准备入睡 |
| **时间点** | 建议执行的时间或时间窗口 | 10:30 前、17:00–18:00、22:30 |
| **剂量** | 建议的量或时长 | 1 杯、30min、20min |

### 4.2 Pin 在时间轴上的展示

- 每个行动建议以**浮动标签**的形式钉在时间轴**右侧**对应时间位置
- Pin 的视觉形式：圆角小标签，内含行为类型图标（☕🏃😴🧘🌙）+ 简短标签（如"喝咖啡"）
- Pin 的纵向位置对应其建议时间，自然嵌入时间线视觉流中——咖啡 pin 在高能区间前，用户直观理解"为什么是这个时间"
- 多个 pin 时间接近时，自动向下排布避免重叠

### 4.3 Pin 的状态

| 状态 | 视觉表现 | 说明 |
|:---|:---|:---|
| 待执行 | 正常颜色 + 实心图标 | 时间点未到 |
| 当前建议 | 高亮 + 脉动效果（可选） | 距最近的下一个建议 |
| 已完成 | 浅色 + ✓ 标记 | 用户已执行（自动检测或手动标记） |
| 已跳过 | 浅色 + 删除线 | 时间已过但未执行 |

### 4.4 Pin 的交互

点击 pin 展开详情浮层（在时间轴上方或下方弹出）：

| 详情内容 | 说明 |
|:---|:---|
| 行为 + 时间 + 剂量 | 完整信息 |
| 原因说明 | 一句话解释（如"10:30 前摄入咖啡因不影响晚间睡眠"） |
| 替代方案 | 可选，如"改为绿茶（咖啡因更低）" |
| 对高能的预估影响 | 可选，如"预计贡献 +0.3h 高能" |
| 进入 Chatflow | "了解更多"按钮 → 进入 Agent 对话 |

### 4.5 建议来源与完成判定

**建议来源**：

| 来源 | 说明 |
|:---|:---|
| 算法自动生成 | 基于用户睡眠数据、生理节律、历史行为模式自动生成当日建议 |
| Agent 对话生成 | 2b（重要日）场景中 Agent 生成的建议，用户一键添加 |
| 用户手动添加 | 未来支持用户自己添加事件 |

**完成判定**：

| 行为类型 | 判定方式 |
|:---|:---|
| 运动 | Apple Health 检测到运动记录 → 自动标记完成 |
| 小睡 | Apple Health 检测到睡眠记录（白天） → 自动标记完成 |
| 咖啡因 | V1 不自动检测，时间过后标记为"已跳过"；用户可手动标记完成 |
| 冥想 | V1 不自动检测，时间过后标记为"已跳过"；用户可手动标记完成 |
| 准备入睡 | 不标记状态，作为时间提示存在 |

> 未来自有硬件和更多数据源接入后，可提升自动检测覆盖率。

## 五、入睡窗口集成

### 5.1 在时间轴上的展示

入睡窗口在时间轴末端以**紫色底色带 + 浮动卡片**展示：

- 位置：时间轴上推荐入睡时间对应位置（底部区域）
- 视觉：紫色半透明背景带 + 🌙 图标 + 入睡窗口卡片（推荐入睡时间 + 明日高能预估）
- 同时在建议列表中有一个 🌙 "准备入睡" 的 pin 标签
- 随一天推进，用户滚动到晚间时入睡窗口自然进入视野

### 5.2 点击进入设置页

点击入睡窗口标记 → 进入睡眠窗口设置页（复用现有 [`sleep-window-prd.md`](../sleep-window/) 定义的独立页面）。

设置页功能不变：
- 圆盘时钟调整入睡/起床时间
- 实时预览明日高能时长变化
- 设置入睡提醒和起床闹钟

### 5.3 与状态 Tab 的联动

用户在入睡窗口设置页调整时间后：
- 状态 Tab 夜间模式的"推荐入睡时间"和"明日高能预估"同步更新
- 今日 Tab 时间轴上的入睡窗口标记位置和预估数值同步更新

## 六、熬夜场景（2c）

**场景描述**：用户今晚需要晚睡，想知道不同入睡时间对明日精力的影响，选择一个损失最小的方案。

**用户流程**：

```
今日 Tab → 点击时间轴上的入睡窗口标记
    ↓
进入睡眠窗口设置页
    ↓ 用户拖动圆盘将入睡时间从 22:30 调到 01:00
    ↓ 实时预览：明日高能从 7.5h 降到 5.8h（-1.7h）
    ↓ 继续尝试 02:00 → 明日高能 4.5h（-3.0h）
    ↓ 回到 01:00（性价比最高）
    ↓
确认今晚入睡窗口为 01:00
    ↓
返回今日 Tab
    ↓ 时间轴上入睡窗口位置更新到 01:00
    ↓ 如果入睡提醒已开启，提醒时间同步更新
```

**体验目标**：务实信任。不说教"你不应该熬夜"，直接展示不同选择的代价（明日高能变化），让用户自己做最优判断。

**设置页要求**：

现有设置页（`sleep-window-prd.md`）已支持拖动调整和实时预览，可完整复用。唯一需要确认的是：
- 当用户选择的入睡时间远晚于推荐时间（如晚 2h 以上），文案上保持"中性呈现差值"原则，不增加负面措辞或警告

## 七、跨 Tab 关联（今日 Tab 侧）

### 7.1 数据一致性

两个 Tab 共享同一套底层数据，以下数据点在今日 Tab 需与状态 Tab 保持一致：

| 数据点 | 状态 Tab | 今日 Tab |
|:---|:---|:---|
| 高能时长 | 第一层核心指标 | 时间轴高能区间的总时长 |
| 高能时段分布 | 第一层时段展示 | 时间轴上的高亮区间 |
| 推荐入睡时间 | 夜间模式第一层（信号灯旁） | 时间轴末端入睡窗口标记 |
| 明日高能预估 | —（v1.2 起夜间模式不展示） | 入睡窗口标记上的预估值 |

当用户在入睡窗口设置页调整时间后，两个 Tab 的相关数据同步更新。

### 7.2 跨 Tab 操作

| 操作 | 起点 | 终点 |
|:---|:---|:---|
| 2b 一键添加行动建议 | 状态 Tab → Chatflow → Agent 生成建议 | 建议作为 pin 添加到今日 Tab 时间轴 |
| 入睡窗口调整 | 今日 Tab → 入睡窗口设置页 → 确认 | 状态 Tab 夜间模式的推荐时间和预估同步更新 |

### 7.3 Smart Digest 归属

- Smart Digest 仅出现在 **状态 Tab**（第二层）
- **今日 Tab 是纯工具型页面，不展示 AI 文字摘要**
- Smart Digest 的快捷提问按钮仅在状态 Tab 展示

### 7.4 与睡眠窗口模块的关系

| 维度 | 今日 Tab 表现 |
|:---|:---|
| 入口 | 今日 Tab 时间轴上的入睡窗口标记 |
| 功能页 | 独立原生页面（参见 `sleep-window-prd.md`） |
| 数据展示 | 状态 Tab 夜间模式 + 今日 Tab 时间轴标记 |
| 设置结果联动 | 同时更新两个 Tab 的相关数据 |

### 7.5 与数据同步状态的关系

- 同步状态条位于 Tab 栏上方，全局指示，不随 Tab 切换变化
- 同步完成后：触发两个 Tab 的所有数据更新
- 同步失败：今日 Tab 展示缓存值

## 八、数据依赖与接口需求

### 8.1 现有接口（可复用）

| 数据 | 来源 | 说明 |
|:---|:---|:---|

```
用户打开 App
    ↓

    ├── /api/today/energy-curve          → 今日 Tab 时间轴
    ├── /api/today/suggestions           → 今日 Tab 行动 pin
    └── /api/sleep-window/init           → 今日 Tab 入睡窗口标记
```

## 九、边界情况与降级（今日 Tab 侧）

### 9.1 数据缺失场景

| # | 场景 | 今日 Tab 表现 |
|:---:|:---|:---|
| 1 | 新用户，无任何数据 | 时间轴仅展示时间刻度，无曲线和 pin，展示引导文案 |
| 2 | 有睡眠数据，无运动数据 | 运动类建议 pin 基于通用推荐 |
| 3 | 数据同步失败 | 展示上次缓存的曲线和建议（参见 `data-sync-status-prd.md`） |
| 4 | 建议接口失败 | 时间轴仅展示曲线和高能区间，无行动 pin |
| 5 | 能量曲线接口失败 | 不展示曲线，仅展示高能时段文字（降级到文字列表） |

### 9.2 时间边界场景

| # | 场景 | 处理方式 |
|:---:|:---|:---|
| 1 | 凌晨 00:00–05:00 打开 | 视为"今晚"，今日 Tab 展示当天数据 |
| 2 | 05:00 后首次打开 | 切换为新一天的时间轴 |
| 3 | 高能时段跨越预计结束时间 | 以实际数据为准更新时间轴 |

### 9.3 交互边界场景

| # | 场景 | 处理方式 |
|:---:|:---|:---|
| 1 | 2b Agent 生成的建议时间已过 | 不添加到时间轴，Agent 提示"这个时间已经过了，建议调整" |
| 2 | 建议 pin 时间冲突 | 视觉上标注时间重叠，不自动合并 |
| 3 | 用户频繁切换 Tab | Tab 内容缓存在内存，切换无加载延迟 |
| 4 | 入睡窗口设置页返回后 | 今日 Tab 数据自动刷新，无需手动下拉 |

## 十、首次引导（今日 Tab 侧）

Onboarding 完成后首次进入首页时触发（仅一次）。在首次引导三步中，今日 Tab 对应第 3 步：

| 步骤 | 聚焦元素 | 说明文案 |
|:---|:---|:---|
| 3 | 今日 Tab 入口 | "切换到「今日」，查看我们为你准备的精力提升建议" |

- 第 3 步完成后自动切换到今日 Tab，引导完成
- 用户可在任意步骤点击"跳过"关闭引导
- 引导完成/跳过状态本地持久化，不再重复触发

## 十一、未来拓展

今日 Tab 的时间轴 + 行动 pin 架构具备良好的拓展性：

| 拓展方向 | 时间线 | 说明 |
|:---|:---|:---|
| **日历同步** | V2 | 用户授权日历后，日程事件显示在时间轴上。系统根据日程自动调整行为建议（如"下午有重要会议，建议 14:00 喝咖啡"） |
| **事件自动检测** | V2 | 运动结束、小睡结束等事件被自动检测后，在时间轴上生成"已完成"的事件 pin，配合高能时长的实时更新形成正反馈 |
| **手动记录** | V2 | 用户可在时间轴上手动添加事件（喝了咖啡、吃了药、喝了酒），系统将其纳入后续建议的调整因素 |
| **药物影响** | V3 | 药物摄入事件可关联药物数据库，预估对能量曲线的影响，在时间轴上可视化展示 |
| **环境数据** | V3 | 自有硬件提供光照、CO₂ 数据后，环境因素作为新的 pin 类型出现（如"现在开窗通风 15min → CO₂ ↓"） |

## 十二、待探讨问题（Open Questions）

> 以下是本次从首页 PRD 中抽离后，需要产品继续对齐细化的关键问题。欢迎在讨论中补充/删除。

### 12.1 核心定位与信息密度

- **Q1**：今日 Tab 是否需要一个"今日摘要卡"（预计高能时长、剩余高能时长、下一个建议）作为首屏固顶？还是严格保持"纯时间轴"？
- **Q2**：用户 70% 的场景只看状态 Tab，今日 Tab 日活如何保障？是否需要推送/徽标驱动用户切过来？
- **Q3**：当天无任何建议 pin 时（数据不足、算法保守），时间轴是否应该降级成其他形态？

### 12.2 时间轴可视化

- **Q4**：每小时 80px、全长 1920px 的纵向轴，是否存在"信息稀释"问题？深夜睡眠时段是否应该可折叠？
- **Q5**：能量曲线的"向右=高能"这一方向，是否有用户测试验证过？横向直方图或左右对称曲线是否更直观？
- **Q6**：高能区间色带 + 能量曲线同时出现时，信息是否冗余？能否二选一？

### 12.3 行动建议事件

- **Q7**：V1 咖啡因/冥想不自动检测 → "已跳过"，是否会给用户带来负反馈？能否默认"已完成"或"未知"？
- **Q8**：2b Agent 一键加入的建议与算法自动生成的建议，在视觉上是否需要区分来源？
- **Q9**：建议 pin 的"剂量"字段（如 30min、1 杯）有多少用户会认真遵守？是否值得占用 pin 详情的展示位？
- **Q10**：pin 时间冲突仅做"视觉标注不自动合并"——如果一天内同时出现 3 个 17:00 的建议，是否会视觉过载？

### 12.4 完成判定与行为闭环

- **Q11**：Apple Health 自动检测完成后，能量曲线是否需要实时刷新以形成正反馈？延迟是多少？
- **Q12**：用户手动标记"已完成"的容忍度——是否需要一个轻交互（长按/滑动）而不是进入详情页？
- **Q13**："已跳过"状态是否需要 recovery 路径（用户后来执行了，能补录吗）？

### 12.5 与状态 Tab 的关系

- **Q14**：状态 Tab 的归因结论中，出现"咖啡喝晚了"这类可行动因子时，能否直接跳转到今日 Tab 对应时间并生成调整建议？
- **Q15**：2b Agent 生成建议 → 添加到今日 Tab 后，用户没有自动切换视角，是否需要 toast / 跳转提示？

### 12.6 熬夜 / 晚起等非常规场景

- **Q16**：用户熬夜到次日 3 点才睡，今日 Tab 的"起点 = 起床前 1h"是否需要动态调整？
- **Q17**：跨时区（用户飞行中）时时间轴的时区归属如何处理？
- **Q18**：倒班 / 非常规作息人群，今日 Tab 是否支持"自定义起止时间"？

### 12.7 指标与成功标准

- **Q19**：今日 Tab 的北极星指标是什么？（建议完成率？Tab 日活？今日 Tab 停留时长？）
- **Q20**：V1 上线后哪些数据需要埋点以验证假设？（pin 点击率、pin 完成率、时间轴滚动深度……）

### 12.8 拓展边界

- **Q21**：手动记录（V2）功能的边界——是否开放到任意自定义事件，还是限定在预设事件类型内？
- **Q22**：日历同步（V2）下，若用户日程与算法建议强冲突，谁优先？是否可以用户偏好配置？
