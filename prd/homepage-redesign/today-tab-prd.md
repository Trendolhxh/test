## 一、背景与定位

### 1.1 在首页双 Tab 中的位置

首页分为两个平级 Tab：

| Tab | 名称 | 核心问题 | 定位 |
|:---|:---|:---|:---|
| 状态 | 状态 | "我怎么样？" | 查看精力状态、归因分析、身体数据 |
| **今日** | 今日 | **"我该做什么？"** | **查看时间轴、行动建议、精力管理** |

- 默认 Tab 为状态 Tab；今日 Tab 是用户主动切换进入
- 切换方式：顶部 Tab 栏点击或左右滑动
- 每次打开 App 始终回到状态 Tab，不记忆上次 Tab

### 1.2 今日 Tab 的产品定位：匹配层

今日 Tab 的本质：

> **把"外部事件"投射到"精力曲线"上，并自动给出调节建议**

它是一个**匹配层**——不生产事件，只做匹配。事件从别处来（算法默认节律、Chatflow 注入，未来是日历、群组），精力曲线一直在那；**匹配和调节建议是这个 Tab 的独有价值**。

这样 V1 → V3 不是功能叠加，而是**事件来源的升级**，闭环本身不变：

| 版本 | 事件来源 | 匹配对象 |
|:---|:---|:---|
| V1 | 算法默认（高能时段、入睡窗口）+ Chatflow 注入 + 种子事件（用户快捷录入） | 个人精力曲线 |
| V2 | + 个人日历同步 | 个人精力曲线 |
| V3 | + 群组协同 | 群体精力曲线交集 |

### 1.3 精力管理的四层用户价值

用户视角下"精力管理"不是一个抽象动作，落到今日 Tab 上是四层具体价值：

| 层 | 用户语义 | 今日 Tab 体现 |
|:---|:---|:---|
| **① 分配** | "把重要的事放进高能时段" | 事件 pin 落在精力曲线上，用户/系统能排 |
| **② 调节** | "用行为改变曲线" | 咖啡 / 运动 / 小睡 / 冥想 / 入睡准备，作为调节类事件 |
| **③ 权衡** | "看到每个选择的代价" | 每个事件可切换"试试另一个"方案，即时预览精力差异 |
| **④ 闭环** | "今天的行为影响明天" | 睡前回看 + 入睡窗口承接，一天的选择延续到次日 |

### 1.4 设计原则（今日 Tab 特有）

| 原则 | 说明 |
|:---|:---|
| **时间即主轴** | 纵向 24h 时间轴是信息组织的核心 |
| **统一事件抽象** | 事件与建议是同一实体，仅 `status` 不同（见第三章） |
| **因果优先于指令** | pin 文案说"为了什么"而非"做什么" |
| **工具型页面** | 纯工具属性，不展示 AI 文字摘要（Smart Digest 仅在状态 Tab） |
| **拓展友好** | 新事件类型、新来源、新群体维度都不改结构，只扩字段值 |

## 二、核心闭环

### 2.1 用户视角的一日四步

```
① 晨起看一眼
   ↓ 打开今日 Tab → 看到精力地图（曲线 + 事件 + 建议）
   ↓ 读懂"今天的独特之处"（见第七章「默认态」顶部总结）
   ↓
② 看系统的匹配结论
   ↓ 每个事件周围已自动配好 1–2 条调节建议
   ↓ 因果明确："10:00 咖啡是为了 11:00 的会"
   ↓
③ 接受 / 忽略 / 试试另一个
   ↓ 默认一切接受；可滑掉某条建议；可"试试另一个"查看代价
   ↓ 想新增主事件 → V1 种子事件快捷录入 / Chatflow；V2 日历
   ↓
④ 白天执行 + 睡前回看
   ↓ 执行：自动检测的类别自动标记完成
   ↓ 回看：事件落在精力曲线的什么位置，建议执行率几何
   ↓ 衔接状态 Tab 夜间模式的入睡准备度
```

### 2.2 闭环在 V1 / V2 / V3 同构

同一个动作形态——"**打开 · 看匹配 · 信或改 · 执行 · 回看**"——在三个版本里都成立，只是事件来源密度和层级变化：

| 版本 | 事件密度 | 用户的核心判断 |
|:---|:---|:---|
| **V1** | 稀疏：系统事件 + Chatflow 注入 + 少量种子事件 | "看懂我今天的节律，接受建议即可" |
| **V2** | 中等：+ 日历日程 | "看懂日程如何踩精力，用建议缓冲" |
| **V3** | 高：+ 群组协同 | "看懂群体交集，和同事协同安排" |

### 2.3 闭环对页面的要求

| 闭环步骤 | 对 UI 的要求 |
|:---|:---|
| ① 晨起看一眼 | 首屏需要"今日特色"一句话（非 LLM 话术，是 data-driven 判断） |
| ② 看匹配结论 | 事件和建议在视觉上主次分明；建议以"为了谁 + 为什么"的因果形式呈现 |
| ③ 接受 / 试试 | 每条建议至少有"忽略"和"试试另一个"两个钩子 |
| ④ 执行 + 回看 | 实时态和回看态视觉区分；自动检测类别回写状态；睡前平滑过渡到入睡准备度 |

## 三、事件实体数据模型

### 3.1 核心抽象：事件与建议合并为同一实体

今日 Tab 上所有"可以出现在时间轴上的东西"——算法推荐的咖啡、用户接受的小睡、日历导入的会议、群组同步的团建——都是**同一种实体**，称为 **Event（事件）**。

**"建议" 只是 `status = suggested` 的事件**。数据模型里不存在"建议 / 事件"两种东西，前端视觉语言也复用同一套 pin。

这样做的收益：

- V1 → V2 → V3 所有扩展在数据上只是 **新增状态、新增来源、新增类别**，不新增实体
- 一条算法建议被用户接受，生命周期是 `suggested → scheduled → in_progress → completed`，不是两个对象之间转移
- 同一套 UI 组件支持所有来源的事件

### 3.2 字段定义（V1）

| 字段 | 类型 | 必填 | 说明 |
|:---|:---|:---:|:---|
| `id` | string | ✓ | 唯一 id |
| `name` | string | ✓ | 名称（如"早会"、"喝咖啡"） |
| `category` | enum | ✓ | 见 3.3 |
| `start_time` | datetime | ✓ | 开始时间 |
| `end_time` | datetime | — | 结束时间，null = 单点时间 |
| `dosage` | string | — | 剂量/强度（1 杯、30min、力量） |
| `status` | enum | ✓ | 见 3.4 |
| `source` | enum | ✓ | 见 3.5 |
| `anchor_id` | string | — | 关联主事件 id（见 3.6） |
| `energy_impact` | object | — | **由外部算法服务返回**，见 3.7 |
| `sleep_window_impact` | object | — | **派生字段，由外部算法服务返回**，见 3.7 |

### 3.3 类别 `category`

V1 共 6 类，V2+ 预留 2 类：

| 值 | 中文 | 示例 |
|:---|:---|:---|
| `sleep` | 睡眠 | 夜间主睡、小睡、午休 |
| `intake` | 摄入 | 咖啡因、酒精、餐、药物 |
| `exercise` | 运动 | 有氧、力量、HIIT |
| `activity` | 日常活动 | 走路、通勤、家务 |
| `regulation` | 调节 | 冥想、呼吸、沐浴、拉伸 |
| `task` | 任务 | 工作、会议、事务 |
| `environment` | 环境（V2+） | 光照、温度、噪音、屏幕 |
| `social` | 社交（V2+） | 饭局、家庭互动 |

**边界**：
- `activity` vs `exercise`：活动 = 日常非运动性身体活动（默认被动产生），运动 = 用户主动安排且有强度概念
- `regulation` vs `exercise`：调节类以"放松/安抚/节律调节"为目的，不用强度评估

### 3.4 状态 `status`

V1 共 5 种，生命周期如下：

```
suggested ─ 用户接受 ─▶ scheduled ─ 当前时间到达 ─▶ in_progress
                                                      │
                            ┌─────────────────────────┼─────────────┐
                            ▼                                       ▼
                        completed                              skipped
                   （自动检测 / 手动标记）                    （时间已过未执行）
```

| 值 | 含义 |
|:---|:---|
| `suggested` | 算法或 Agent 生成的建议，待用户决定 |
| `scheduled` | 已确认（用户接受 / 日历导入 / 用户手动添加 / 种子事件录入） |
| `in_progress` | 当前时间落在 `start_time / end_time` 区间内 |
| `completed` | 已完成（Apple Health 检测或用户手动标记） |
| `skipped` | 已跳过（时间已过但未完成） |

### 3.5 来源 `source`

| 值 | 含义 | V1 | V2 | V3 |
|:---|:---|:---:|:---:|:---:|
| `system` | 算法默认（高能时段、入睡窗口、通用调节建议） | ✓ | ✓ | ✓ |
| `agent` | Chatflow 中 Agent 生成（2b） | ✓ | ✓ | ✓ |
| `user` | 用户手动录入（V1 通过"种子事件"快捷入口有限开放） | 🔶 | ✓ | ✓ |
| `calendar` | 个人日历同步 |  | ✓ | ✓ |
| `group` | 群组协同 |  |  | ✓ |

> V1 标 🔶：仅支持"种子事件"快捷录入的 4 个预设类型（见第七章 7.4），不支持任意事件手动录入。

### 3.6 关联事件 `anchor_id`

`anchor_id` 连起"主事件 — 辅助建议"的因果链：

- **辅助建议**（`status=suggested` 且 `source ∈ {system, agent}`）**必须**指向一个 `anchor_id`——它是为了某个主事件服务的
- **主事件**（任务、会议、运动本身、睡眠）可以无 `anchor_id`，独立存在
- 时间轴 UI 上应当体现这种主次关系（见第六章 6.5）

**示例**：

```
任务 11:00 "产品评审"         ← anchor 为空
   └─ 建议 10:00 "喝咖啡"      ← anchor_id = 任务 id，为它服务
   └─ 建议 14:00 "小睡 20min"  ← anchor_id = 任务 id，为它的余波恢复服务
```

### 3.7 精力影响与入睡窗口影响

`energy_impact` 和 `sleep_window_impact` 是**外部统一算法服务**的输出，今日 Tab 只消费、不计算。

**算法策略**：冷启动阶段使用内置映射表（如"咖啡因 200mg × 上午 → +0.5h 高能"），积累用户历史后切换到**用户个人回归模型**（方案 C：映射表 + 学习的混合策略）。

**本 PRD 只定义字段契约，不定义算法细节**。具体算法策略由"精力算法服务"独立文档描述。

字段契约（占位设计，后续与算法服务对齐）：

```
energy_impact: {
  delta_h: number         // 对当日总高能时长的净贡献（小时）
  curve_adjustment: [...] // 对未来精力曲线的逐点调整（可选）
  ...
}

sleep_window_impact: {
  bedtime_shift_min: number   // 对推荐入睡时间的偏移（分钟）
  tomorrow_energy_delta_h: number  // 对明日高能时长的预估影响
}
```

## 五、纵向 24h 能量时间轴

### 5.1 核心定义

一条从上到下覆盖 24 小时的**纵向可滚动时间轴**，是今日 Tab 的视觉主体。用户通过上下滑动浏览全天。

**布局结构**：

| 要素 | 说明 |
|:---|:---|
| 时间轴方向 | **纵向**，时间从上到下流动 |
| 起点 | 用户常规起床时间前 1h（如常规 06:30 起床，从 05:30 开始） |
| 终点 | 次日同一时间（覆盖完整 24h + 入睡时段） |
| 左侧时间标签 | 每小时一个标签（05、06、07…00、01），固定在时间轴左侧边缘 |
| 中央区域 | **连续时间线**：无区块分割，以能量曲线 + 背景色带展示全天能量变化 |
| 右侧 | **行动 pin**：以浮动标签形式钉在对应时间位置 |
| 能量曲线 | 贯穿全天的纵向贝塞尔曲线（X=能量水平，Y=时间），向右=高能，向左=低能 |

### 5.2 视觉分层

时间轴采用**连续时间线 + 背景色带高亮**的方式，不划分独立区块：

| 视觉层 | 说明 |
|:---|:---|
| **时间刻度** | 每小时一条水平参考线 + 左侧时间标签 |
| **睡眠区底色** | 紫色半透明底色覆盖昨夜睡眠时段（0:00 → 起床时间），叠加睡眠时长和效率文字 |
| **高能时段底色** | **品牌绿色半透明底色**覆盖高能窗口区间，左上角标注 ⚡ 高能 + 起止时间 |
| **能量曲线** | 一条贯穿全天的纵向贝塞尔曲线，展示能量水平的连续变化（详见 3.4） |
| **入睡窗口** | 紫色半透明底色 + 🌙 入睡窗口卡片（推荐入睡时间 + 明日预估高能） |
| **行动 pin** | 浮动标签钉在右侧对应时间位置（详见 §四） |

### 5.3 时间轴的三种状态

**预测态**（晨起，当日首次打开）：

| 内容 | 说明 |
|:---|:---|
| 时间轴 | 全天数据均为预测，统一样式展示 |
| 能量曲线 | 全天曲线统一为虚线（`stroke-dasharray`），透明度降低 |
| 行动 pin | 所有当日建议 pin 可见，状态为"待执行" |
| 自动定位 | 滚动到起床时间附近 |

**实时进度态**（日间打开）：

| 内容 | 说明 |
|:---|:---|
| 当前时间标记 | 一条横线 + 红色圆点 + "现在" 标签，贯穿时间轴宽度 |
| 上方（已过） | 曲线以实线展示（实际数据） |
| 下方（未来） | 曲线以虚线展示（预测数据），通过 SVG clipPath 分割 |
| 行动 pin | 已过的标记完成/跳过，未来的正常展示 |
| 自动定位 | 滚动到当前时间标记附近 |

**回顾态**（睡前/夜间打开）：

| 内容 | 说明 |
|:---|:---|
| 时间轴 | 全天均为实际数据 |
| 能量曲线 | 全天实线展示 |
| 行动 pin | 全部标记为已完成/已跳过 |
| 入睡窗口 | 底部入睡窗口卡片突出展示 |
| 自动定位 | 滚动到晚间入睡窗口附近 |

### 5.4 全天连续能量曲线

一条**贯穿全天**的纵向能量曲线，覆盖从 0 时到入睡窗口后 1h 的完整时间范围（不再仅限于高能区块内部）：

- 使用 Catmull-Rom 样条插值 → 三次贝塞尔 SVG 路径，不使用折线
- Y 轴 = 时间（从上到下流动），X 轴 = 能量水平（向右 = 高能）
- 曲线左侧（曲线 → 左边界）以品牌绿色半透明填充，形成面积图效果
- 睡眠时段能量值极低（曲线贴近左边界），起床后逐渐上升，高能窗口时达到峰值
- 实时态下：当前时间线以上为实线（实际数据），以下为虚线（预测数据），通过 SVG clipPath 实现

### 5.5 可视化要求

| 要求 | 说明 |
|:---|:---|
| 曲线平滑 | 使用贝塞尔曲线或样条插值，不使用折线 |
| 滚动交互 | 支持**垂直滑动**浏览全天；自动定位到当前时间/关键区域 |
| 每小时高度 | 约 80px，整个 24h 时间轴总高度约 1920px，需要滚动查看 |
| 性能 | 区块和曲线渲染 ≤ 1 帧（16ms），滚动流畅 60fps |
| 响应式 | 适配屏幕宽度，时间轴占满 Tab 内容区宽度 |

## 六、事件在时间轴上的展示

> 本章描述事件（Event 实体，见第三章）在时间轴上的视觉与交互。所有事件——无论 `suggested` 还是 `scheduled`，无论来自 `system` 还是 `user`——复用同一套 pin 语言。

### 6.1 类别与图标

| `category` | 图标 | 示例 |
|:---|:---:|:---|
| `sleep` | 😴 | 小睡、午休 |
| `intake` | ☕ / 🍽️ / 🍷 / 💊 | 咖啡、餐、酒、药 |
| `exercise` | 🏃 | 有氧、力量 |
| `activity` | 🚶 | 走路、通勤 |
| `regulation` | 🧘 | 冥想、呼吸、沐浴 |
| `task` | 📌 | 工作、会议 |
| 入睡窗口（特殊系统事件） | 🌙 | 准备入睡 |

### 6.2 Pin 在时间轴上的展示

- 每个事件以**浮动标签（pin）** 的形式钉在时间轴**右侧**对应时间位置
- 视觉形式：圆角小标签，内含 `category` 图标 + 时间 + 主文案（因果式，见 7.4）
- 纵向位置对应 `start_time`；多个 pin 时间接近时自动向下排布避免重叠
- 跨时间段的事件（如 17:00–18:00 运动）用带细条的 pin 延展展示

### 6.3 Pin 的视觉状态（对齐 `status`）

| `status` | 视觉表现 |
|:---|:---|
| `suggested` | 正常颜色 + 实心图标 + 虚线描边（暗示"待决定"） |
| `scheduled` | 正常颜色 + 实心图标 + 实线描边（已确认） |
| `in_progress` | 高亮 + 脉动效果 |
| `completed` | 浅色 + ✓ 标记 |
| `skipped` | 浅色 + 删除线 |

### 6.4 Pin 的交互

点击 pin 展开详情浮层，内容随 `status` 和 `source` 自适应：

| 元素 | 说明 |
|:---|:---|
| 名称 + 时间 + 剂量 | 事件基本信息 |
| 因果一句话 | 主文案展开版（如"10:30 前摄入咖啡因不影响晚间睡眠"） |
| 关联主事件 | 若 `anchor_id` 非空，点击可跳转到该主事件 |
| 精力影响预估 | 展示 `energy_impact.delta_h`（由算法服务返回） |
| 操作按钮（suggested） | "接受 / 忽略 / 试试另一个"（见 7.5） |
| 操作按钮（scheduled） | "取消 / 编辑时间" |
| 操作按钮（其他状态） | 仅展示，不可操作 |
| 进入 Chatflow | "了解更多"按钮 → 进入 Agent 对话 |

### 6.5 主事件 — 辅助建议的视觉绑定

当一个主事件（如 11:00 产品评审）周围存在多条辅助建议（如 10:00 咖啡、14:00 小睡）时，UI 需要让用户**一眼看出它们属于同一个因果簇**：

- **方案 A**（V1 推荐）：辅助建议 pin 的左侧出现一条细线延伸到主事件，形成"分支"视觉
- **方案 B**：主事件被点击时，相关辅助建议高亮；否则弱化
- **方案 C**：主事件 pin 尺寸略大，辅助 pin 略小

V1 建议采用 A + C 的组合。

### 6.6 完成判定

| `category` | 自动完成判定 |
|:---|:---|
| `exercise` | Apple Health 检测到运动记录 → 自动 `completed` |
| `sleep`（小睡） | Apple Health 检测到日间睡眠 → 自动 `completed` |
| `intake`（咖啡、酒、餐） | V1 不自动检测；时间过后 `skipped`；用户可手动 `completed` |
| `regulation` | V1 不自动检测；时间过后 `skipped`；用户可手动 `completed` |
| `task` | V1 不标记完成；时间过后 pin 自动弱化 |
| `activity` | 被动由 Apple Health 活动数据填充（不走 suggested→completed 流程） |

> V2+ 自有硬件和更多数据源接入后，可提升自动检测覆盖率（如咖啡因识别、HRV 对冥想的回应判断）。

## 七、默认态设计

### 7.1 问题定义

V1 里用户事件来源只有 `system` 和 `agent`，其中 `agent` 还需要用户主动去 Chatflow 触发。所以**大多数用户打开今日 Tab 时，默认态上只有 system 生成的东西**——高能时段、入睡窗口 + 4 条通用建议（咖啡、运动、小睡、准备入睡）。

这个默认态当前存在的问题：

| 问题 | 用户感受 |
|:---|:---|
| 建议通用化——咖啡/运动/小睡是普适健康常识 | "这不是我每天听过的话吗" |
| 看了没有动作——没东西可点、可改、会变 | 静态宣读，不是工具 |
| 没有"今天"的特异性——看不出今天和昨天/平常有何不同 | "哦" 就关掉 |

默认态要回答两个让用户想继续用的问题：

> **Q1**：今天我的精力地图有什么**特别之处**？
> **Q2**：我现在可以**马上做一个小动作**改变它吗？

### 7.2 三层手法

默认态通过三层组合拳，把"静态板"升级为"最小可玩循环"：

| 层 | 目标 | 具体手法 |
|:---|:---|:---|
| **第一层** | 让用户看到今天的独特性 | 顶部"今日特色"结构化总结 |
| **第二层** | 让建议说出因果 | 事件 pin 文案因果化 |
| **第三层** | 让用户能动一下 | "试试另一个" + "种子事件"录入口 |

### 7.3 第一层：今日特色总结

在时间轴顶部（首屏可见位置），用一句 **data-driven 的结构化判断**总结今天的独特之处。

**特点**：
- 不是 LLM 生成的话术（状态 Tab 的 Smart Digest 已经承担）
- 不是模板占位（"今天状态不错"这种）
- 是**从曲线 + 事件 + 历史数据里自动提取的事实 + 可行动含义**

**文案库（示例）**：

| 触发条件 | 总结文案 |
|:---|:---|
| 高能区间起点比近 7 天平均早 ≥30min | "今天你的高能开得比平常早，抓紧上午" |
| 无晚间高能窗口 | "今天没有晚间高能段，18 点后精力会下滑" |
| 两高能段之间低谷 ≤ 30% 峰值 | "13–15 点是今天的低谷期，安排轻任务" |
| 总高能时长比近 7 天均值少 ≥1h | "今天高能比平常少 1 小时，关键事早点做" |
| 总高能时长比近 7 天均值多 ≥1h | "今天高能比平常多 1 小时，难事留给你" |
| 入睡窗口比平常早 ≥30min | "今天需要早点入睡，调整晚间节奏" |
| 未命中任何特殊规则 | 不展示此条，避免空话 |

**视觉**：
- 位置：时间轴顶部固定，占一行
- 样式：品牌色左侧色条 + 14px 文字，不抢主视觉
- 不可点击（V1），后续可接入 Chatflow（"为什么今天这样？"）

**降级**：数据不足时（新用户前 3 天、同步失败）此条不展示，时间轴直接从顶部开始。

### 7.4 第二层：事件 pin 文案因果化

当前 pin 文案是**指令式**（"10:30 喝咖啡"），改为**因果式**（"为了什么"）。

| 当前（指令式） | 改版（因果式） |
|:---|:---|
| ☕ 10:30 喝咖啡 | ☕ 10:30 前 · 保住上午高能 |
| 😴 13:00 小睡 20min | 😴 13:00 · 填平下午低谷 |
| 🏃 17:00 运动 | 🏃 17:00 · 延长晚间高能 |
| 🌙 22:30 准备入睡 | 🌙 22:30 · 保住明日高能 |

**规则**：
- Pin 主标签 = 图标 + 时间 + "为什么"一句话（不超过 10 字）
- 展开详情浮层才出现具体行为名、剂量、替代方案
- 当 `anchor_id` 非空时，"为什么"优先引用 anchor 事件（"为了 11:00 的会议"）

### 7.5 第三层（A）：每条建议的"试试另一个"钩子

每个 `status=suggested` 的事件在详情浮层里除了"接受 / 忽略"，多一个 **"试试另一个"** 按钮。点击后展示对比预览：

```
☕ 当前建议：10:00 喝咖啡
   → 上午高能 +0.3h，不影响晚间入睡

其他方案：
─────────────────────────
○ 不喝咖啡           → 上午高能 持平
                       晚间高能 持平
─────────────────────────
○ 改喝绿茶           → 上午高能 +0.1h
                       晚间高能 持平
─────────────────────────
○ 14:00 再喝         → 上午高能 持平
                       晚间入睡延迟 30min（不建议）
─────────────────────────
```

**V1 简化版**：对比选项由算法服务返回的 `alternatives` 字段提供（V1 先支持咖啡、运动、小睡三类，其他类别仅展示"忽略/接受"）。

**目的**：让用户第一次感受到"**这些建议不是死的，是可以换方案的**"——从浏览升级为管理。

### 7.6 第三层（B）：今日安排描述（吸底输入）

把"引导用户主动描述今日安排 → 生成辅助建议"的能力放在**页面底部吸底交互区**，替代 v0.2 中顶部 "+" 种子事件入口的位置。

#### 7.6.1 设计理念

V1 里今日 Tab 是**工具**不是**聊天**。底部交互区承担唯一的"告诉系统今天什么特别"输入通道：
- **结构化输入**（非自由对话）—— 明确和状态 Tab 的 Smart Digest 边界：今日 Tab **不展示 AI 文字摘要，也不提供自由对话入口**
- **说完即走** —— 用户描述 → 算法生成 → 用户勾选接受 → 建议落到时间轴，路径单向、终点明确
- **键盘 + 语音** —— 移动端优先，语音输入覆盖手忙 / 开车 / 懒于打字等场景

#### 7.6.2 吸底条默认态

两行吸底条常驻今日 Tab 底部：
- 第 1 行：输入框（引导文案「今天有什么特别安排？」）+ 右侧麦克风图标（语音输入）
- 第 2 行：预设 chips：`和昨天一样` / `和上周 {weekday} 一样`
  - weekday 默认同今天（今天周四 → "和上周四一样"），V1 不支持选其他日
  - **无历史数据时 chips 不显示**（新用户首日、无昨天事件、无上周同 weekday 数据），只保留输入框引导

点击输入框或 chip → 半层从底部上拉。

#### 7.6.3 半层三态

**输入态**：
- 上拉至约 60% 屏高，键盘自动弹起
- 输入框 + chips（同吸底条），chip 点击直接触发生成
- 顶部拖拽条 / "取消" 收回至吸底默认态

**生成中态**：
- 半层内显示"正在为你生成建议…"的 loading
- **允许用户下拉收起半层**、继续浏览时间轴；收起后吸底条切换为"生成中"态
- 若算法 timeout：半层显示"生成失败 · 重试"

**确认态**（生成完毕）：
- 建议列表 —— 每条：时间、图标、label、因果短语（见 7.4）、能量影响数值
- **默认全部勾选**，顶部提供 `[全不选]` 快捷
- 底部主按钮 `[接受勾选的 N 条]`
- 次按钮 `[重新描述]`（返回输入态，清空当前建议列表）
- 次按钮 `[都不要]`（关闭半层，建议全部丢弃）

#### 7.6.4 吸底条两态（半层收起时的状态显示）

| 态 | 触发 | 显示 | 点击行为 |
|:---|:---|:---|:---|
| 生成中 | 用户在生成过程中下拉收起半层 | "⏳ 正在生成建议..." | 拉起半层至生成中态 |
| 生成完毕 | 生成完成 & 半层已收起 & 用户未处理 | "✨ 建议已生成 · N 条 [查看]" | 拉起半层至确认态 |

**关键约束**：生成完毕的吸底条**一直留着**，直到用户完成"接受 / 都不要 / 重新描述"三种终态之一。**不自动消失、不自动丢弃**。

#### 7.6.5 接受流程

用户点 `[接受勾选的 N 条]` → 勾选项 `status: suggested → scheduled` → 以动画落位到时间轴对应 pin 位置 → 半层收起 → 吸底条回到默认态。

**时间冲突不做处理**：建议 14:30 喝咖啡、但用户 14:30 已有会议 —— 照常落位，由用户在时间轴上长按 pin 微调。

#### 7.6.6 二次输入

用户当天已输入过一次，之后再拉起半层：**空白重开**。
- 不回显之前输入的文字
- 不回显之前生成的建议列表
- 新生成的建议**仅影响未来时段**：`now()` 之后的时段可被新建议占位；`now()` 之前的时段（已完成 / 已跳过 / 实际发生的事件）**不被新建议变更**

**数据落地**：所有被接受的建议 `source=user`（表示用户描述触发），`status=scheduled`。

### 7.7 默认态组合效果

三层手法合在一起，默认态从"**读一下然后关掉**"升级为"**看懂今天特点 → 懂为什么这么建议 → 告诉系统今天安排 → 换/接受建议**"的**最小可玩循环**。

```
打开今日 Tab
    ↓
看到顶部："今天没有晚间高能段，18 点后精力会下滑"
    ↓
看到 pin："🏃 17:00 · 延长晚间高能"，觉得不合适
    ↓
点击 pin → "试试另一个" → 看到"不运动 → 晚间平稳无增"
    ↓ 接受换方案，或者
    ↓
从底部吸底条点 [和昨天一样]，或在输入框说/写
  "下午 3 点有重要会议"
    ↓
半层弹起，进入生成中态
    ↓ 用户下拉收起半层继续浏览时间轴
    ↓ 吸底条显示 "⏳ 正在生成建议..."
    ↓
生成完毕，吸底条变 "✨ 建议已生成 · 3 条 [查看]"
    ↓
用户点击 → 拉起半层至确认态
    ↓ 3 条建议全部默认勾选
    ↓ 点 [接受勾选的 3 条]
    ↓
时间轴上新增 3 个 pin 动画落位：
  14:30 喝咖啡 · 保住会议前高能
  13:00 小睡 20 分 · 填平低谷
  17:30 散步 · 延长晚间高能
    ↓
用户第一次感到"我在和系统合作管理精力"
```

## 八、入睡窗口集成

### 8.1 在时间轴上的展示

入睡窗口在时间轴末端以**紫色底色带 + 浮动卡片**展示：

- 位置：时间轴上推荐入睡时间对应位置（底部区域）

- 同时在建议列表中有一个 🌙 "准备入睡" 的 pin 标签

### 8.2 点击进入设置页

点击入睡窗口标记 → 进入睡眠窗口设置页（复用现有 [`sleep-window-prd.md`](../sleep-window/) 定义的独立页面）。

设置页功能不变：
- 圆盘时钟调整入睡/起床时间
- 实时预览明日高能时长变化
- 设置入睡提醒和起床闹钟

### 8.3 与状态 Tab 的联动

用户在入睡窗口设置页调整时间后：
- 状态 Tab 夜间模式的"推荐入睡时间"和"明日高能预估"同步更新
- 今日 Tab 时间轴上的入睡窗口标记位置和预估数值同步更新

## 九、熬夜场景（2c）

**场景描述**：用户今晚需要晚睡，想知道不同入睡时间对明日精力的影响，选择一个损失最小的方案。

**用户流程**：

```
今日 Tab → 点击时间轴上的入睡窗口标记
    ↓
进入睡眠窗口设置页
    ↓ 用户拖动圆盘将入睡时间从 22:30 调到 01:00
    ↓ 实时预览：明日高能从 7.5h 降到 5.8h（-1.7h）
    ↓ 继续尝试 02:00 → 明日高能 4.5h（-3.0h）
    ↓ 回到 01:00（性价比最高）
    ↓
确认今晚入睡窗口为 01:00
    ↓
返回今日 Tab
    ↓ 时间轴上入睡窗口位置更新到 01:00
    ↓ 如果入睡提醒已开启，提醒时间同步更新
```

**体验目标**：务实信任。不说教"你不应该熬夜"，直接展示不同选择的代价（明日高能变化），让用户自己做最优判断。

**设置页要求**：

现有设置页（`sleep-window-prd.md`）已支持拖动调整和实时预览，可完整复用。唯一需要确认的是：
- 当用户选择的入睡时间远晚于推荐时间（如晚 2h 以上），文案上保持"中性呈现差值"原则，不增加负面措辞或警告

## 十、跨 Tab 关联（今日 Tab 侧）

### 10.1 数据一致性

两个 Tab 共享同一套底层数据，以下数据点在今日 Tab 需与状态 Tab 保持一致：

| 数据点 | 状态 Tab | 今日 Tab |
|:---|:---|:---|
| 高能时长 | 第一层核心指标 | 时间轴高能区间的总时长 |
| 高能时段分布 | 第一层时段展示 | 时间轴上的高亮区间 |
| 推荐入睡时间 | 夜间模式第一层（信号灯旁） | 时间轴末端入睡窗口标记 |
| 明日高能预估 | —（v1.2 起夜间模式不展示） | 入睡窗口标记上的预估值 |

当用户在入睡窗口设置页调整时间后，两个 Tab 的相关数据同步更新。

### 10.2 跨 Tab 操作

| 操作 | 起点 | 终点 |
|:---|:---|:---|
| 2b 一键添加行动建议 | 状态 Tab → Chatflow → Agent 生成建议 | 建议作为 pin 添加到今日 Tab 时间轴 |
| 入睡窗口调整 | 今日 Tab → 入睡窗口设置页 → 确认 | 状态 Tab 夜间模式的推荐时间和预估同步更新 |

### 10.3 Smart Digest 归属 / 对话入口边界

- Smart Digest 仅出现在 **状态 Tab**（第二层）
- **今日 Tab 是纯工具型页面**：不展示 AI 文字摘要，**也不提供自由对话聊天入口**
- 今日 Tab 底部吸底交互区（见 7.6）**仅支持结构化输入**（描述今日安排 → 算法生成辅助建议），语义、格式和 Smart Digest 的自由对话明确区分
- Smart Digest 的快捷提问按钮仅在状态 Tab 展示

### 10.4 与睡眠窗口模块的关系

| 维度 | 今日 Tab 表现 |
|:---|:---|
| 入口 | 今日 Tab 时间轴上的入睡窗口标记 |
| 功能页 | 独立原生页面（参见 `sleep-window-prd.md`） |
| 数据展示 | 状态 Tab 夜间模式 + 今日 Tab 时间轴标记 |
| 设置结果联动 | 同时更新两个 Tab 的相关数据 |

### 10.5 与数据同步状态的关系

- 同步状态条位于 Tab 栏上方，全局指示，不随 Tab 切换变化
- 同步完成后：触发两个 Tab 的所有数据更新
- 同步失败：今日 Tab 展示缓存值

## 十二、边界情况与降级（今日 Tab 侧）

### 12.1 数据缺失场景

| # | 场景 | 今日 Tab 表现 |
|:---:|:---|:---|
| 1 | 新用户，无任何数据 | 时间轴仅展示时间刻度，无曲线和 pin，展示引导文案 |
| 2 | 有睡眠数据，无运动数据 | 运动类建议 pin 基于通用推荐 |
| 3 | 数据同步失败 | 展示上次缓存的曲线和建议（参见 `data-sync-status-prd.md`） |
| 4 | 建议接口失败 | 时间轴仅展示曲线和高能区间，无行动 pin |
| 5 | 能量曲线接口失败 | 不展示曲线，仅展示高能时段文字（降级到文字列表） |

### 12.2 时间边界场景

| # | 场景 | 处理方式 |
|:---:|:---|:---|
| 1 | 凌晨 00:00–05:00 打开 | 视为"今晚"，今日 Tab 展示当天数据 |
| 2 | 05:00 后首次打开 | 切换为新一天的时间轴 |
| 3 | 高能时段跨越预计结束时间 | 以实际数据为准更新时间轴 |

### 12.3 交互边界场景

| # | 场景 | 处理方式 |
|:---:|:---|:---|
| 1 | 2b Agent 生成的建议时间已过 | 不添加到时间轴，Agent 提示"这个时间已经过了，建议调整" |
| 2 | 建议 pin 时间冲突 | 视觉上标注时间重叠，不自动合并 |
| 3 | 用户频繁切换 Tab | Tab 内容缓存在内存，切换无加载延迟 |
| 4 | 入睡窗口设置页返回后 | 今日 Tab 数据自动刷新，无需手动下拉 |

## 十三、首次引导（今日 Tab 侧）

Onboarding 完成后首次进入首页时触发（仅一次）。在首次引导三步中，今日 Tab 对应第 3 步：

| 步骤 | 聚焦元素 | 说明文案 |
|:---|:---|:---|
| 3 | 今日 Tab 入口 | "切换到「今日」，查看我们为你准备的精力提升建议" |

- 第 3 步完成后自动切换到今日 Tab，引导完成
- 用户可在任意步骤点击"跳过"关闭引导
- 引导完成/跳过状态本地持久化，不再重复触发

## 十四、未来拓展

今日 Tab 的时间轴 + 行动 pin 架构具备良好的拓展性：

| 拓展方向 | 时间线 | 说明 |
|:---|:---|:---|
| **日历同步** | V2 | 用户授权日历后，日程事件显示在时间轴上。系统根据日程自动调整行为建议（如"下午有重要会议，建议 14:00 喝咖啡"） |
| **事件自动检测** | V2 | 运动结束、小睡结束等事件被自动检测后，在时间轴上生成"已完成"的事件 pin，配合高能时长的实时更新形成正反馈 |
| **手动记录** | V2 | 用户可在时间轴上手动添加事件（喝了咖啡、吃了药、喝了酒），系统将其纳入后续建议的调整因素 |
| **药物影响** | V3 | 药物摄入事件可关联药物数据库，预估对能量曲线的影响，在时间轴上可视化展示 |
| **环境数据** | V3 | 自有硬件提供光照、CO₂ 数据后，环境因素作为新的 pin 类型出现（如"现在开窗通风 15min → CO₂ ↓"） |
